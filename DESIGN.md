# create-scrum-team-workspace 设计说明

## 目标

从零生成一个可用于 Scrum 团队协同开发的项目工作区，让团队启动时同时拥有：

- 项目运行层：产品发现、Backlog、Sprint、风险、会议、度量和知识库。
- 工程设计层：ADR、API、数据模型、前端设计系统、技术债和 Spike。
- 代码落地层：轻量代码仓库骨架、`TeamWork/` 协同目录、分支命名和 Git 身份配置说明。
- 角色协同层：角色表、角色卡、能力矩阵、备份机制、Sprint 0 分工、SM 作战手册和可配置命名系统。

## 项目类型

| type | 适用场景 | 生成后的第一动作 |
| --- | --- | --- |
| `new` | 从零新项目 | 填愿景、建首批 Backlog、确认工程技术栈 |
| `legacy` | 存量项目重构 | 做现状诊断、冻结参考源、确定迁移边界 |
| `product` | 成熟产品迭代 | 梳理 Backlog、容量规划、补齐质量门禁 |
| `prototype` | 原型转正 | 复核技术选型、重建目录、补齐安全和测试基线 |

## 角色系统

固定角色槽位保持 Scrum 与工程职责稳定：

| 槽位 | 默认职责 |
| --- | --- |
| `po` | 产品负责人 / PM |
| `sm` | Scrum Master |
| `tl` | TL / Sr.BE |
| `midbe` | Mid.BE / QA API |
| `srfe` | Sr.FE / UX |
| `midfe` | Mid.FE / QA E2E |
| `fs` | Full-stack / DevOps |

命名系统只改变“显示名称”和工具链占位，不改变职责槽位。这样既能保留团队氛围，也能保证文档、分支、目录和 Git 身份可追溯。

## CLI 能力

```bash
node index.mjs <project-name> \
  --type=new|legacy|product|prototype \
  --preset=tech|myth|wuxia|compass|studio|greek \
  --repo=<repo-name> \
  --role.midfe=Aurora \
  --email.po=jobs@acme.com \
  --config=./scrum.config.json \
  --interactive \
  --dry-run \
  --git-root=repo|workspace|none \
  --worktrees \
  --role-test-commits \
  --remote=<url> \
  --push \
  --sprint=1 \
  --no-git
```

### 交互流程

`--interactive`（或缺省项目名时自动触发）会逐项询问项目、仓库和角色信息。角色套装选定后先展示全量角色表，用户可按槽位或全部调整名称与邮箱；随后确认 worktree、角色测试提交、远端和推送策略，最后在**摘要确认**页输入 Y 才会写盘。

### 占位符约定

- 文件内容统一使用 `{{KEY}}` 占位符（如 `{{PROJECT_NAME}}`、`{{REPO_NAME}}`、`{{CREATED_DATE}}`）。
- 文件名/目录名使用 `__KEY__` 形式（如 `__PROJECT_NAME__.code-workspace`、`10_代码仓库/__REPO_NAME__/`），避免与文档正文里的普通词产生冲突。

## 取舍

- 保持零依赖，便于复制、审阅和本地执行。
- 不直接生成具体技术栈代码，避免把模板绑定到某个框架。
- 项目工作区文档是主事实源，代码仓库不再生成重复的 `01-docs` 文档中心；代码仓库只保留 `apps/`、`infra/`、README、可执行资产和链接。
- 生成文档多于生成业务代码，因为 Scrum 团队协同的最大风险通常是输入输出、角色边界、证据链和集成节奏不清。
- 默认使用 `feature/sprint-<n>/<topic>-<name>-<role>` 分支名，目录名允许显示名称但会清理明显危险字符。
- 默认使用 `repo` 模式，让 5 个角色 worktree 只检出代码仓内容；`workspace` 作为早期单仓选项保留。
- 远端配置和推送是两个动作，避免提供 URL 后意外写入远端。
- 角色就绪提交是可选验证证据，只存在于各自分支，不自动合入主线。

## 演进决策

完整变更列表见 `CHANGELOG.md`。下表只回答"为什么这么做"，作为后续维护者的参考。

| 版本 | 决策 | 起因 | 选项与取舍 | 结论 |
| --- | --- | --- | --- | --- |
| 0.3.3 | Sprint 流程监控台替代工作进度表 | 原表只记录成员进度，无法回答阶段、门禁和角色下一行动；另建总表又会重复事实 | A. 扩大输入输出总表；B. 新增第二张总表；C. 原位升级 Sprint 进度表 | 选 C；总表管产物事实，监控台管当前流动 |
| 0.3.3 | 阶段采用“主阶段 + 工作流阶段”双轴表达 | Scrum 中产品、工程、质量和发布常并行推进，单一阶段会误导等待和开工判断 | A. 全团队单阶段；B. 每角色自由描述；C. 主阶段配并行工作流门禁 | 选 C，既保持全局方向又反映真实并行状态 |
| 0.3.3 | 角色行动统一分为五类 | “下一步做什么”若只有待办/阻塞，无法区分等待、可逆准备、清障和升级边界 | A. 仅任务状态；B. 五类行动分类 | 选 B：必做、等待输入、可提前先行、协作清障、暂停/升级 |
| 0.3.0 | 默认独立代码仓并创建 5 个角色 worktree | 编码角色和职责已知，手工复制/配置既慢又易串号 | A. 只输出命令；B. 自动创建 clone；C. 自动创建 worktree | 选 C，共享对象库且目录隔离；PO/SM 不创建编码目录 |
| 0.3.0 | Git 身份使用 worktree config | 普通 local config 被所有 worktree 共享，角色身份会互相覆盖 | A. 每人独立 clone；B. 环境变量提交；C. `extensions.worktreeConfig` | 选 C，保留 worktree 优势并真正隔离身份 |
| 0.3.0 | 个人分支改为 `feature/sprint-n/...` | Git 不能同时保存 `sprint-n` 与 `sprint-n/...` 引用 | A. 改集成分支；B. 改个人分支命名空间 | 选 B，保留团队熟悉的 `sprint-n` 集成分支 |
| 0.3.0 | 远端配置与 push 分离 | URL 可能填错，远端也可能已有历史 | A. 有 URL 就 push；B. 显式 `--push` | 选 B，且永不自动 force |
| 0.3.0 | 角色测试提交为可选且不合入 | 需要验证提交身份，但无业务价值的提交不应污染主线 | A. 全部合入；B. 只留个人分支；C. 不提供 | 选 B |
| 0.2.0 | 路径占位符统一为 `__KEY__` | 原 `PROJECT_NAME` 裸串替换缺乏自描述性，未来新增需要进路径的占位符容易踩坑 | A. 保持现状；B. 改 `{{KEY}}` 但与 NTFS/部分工具不友好；C. 改 `__KEY__` | 选 C，内容用 `{{KEY}}`、路径用 `__KEY__`，形成对称的两套规则 |
| 0.2.0 | 模板拷贝两阶段化（plan → apply） | 原一次性写盘无法 dry-run，失败时残留半个工作区 | A. 单阶段加 try/finally 回滚；B. 拷到临时目录再 rename；C. 计划/执行分离 | 选 C，同时获得 dry-run 能力且实现最简 |
| 0.2.0 | 新增 `--config` JSON 配置 | 交互模式适合首次，重建/对比实验需要可重放 | A. 共享 shell 别名脚本；B. 命令行长串组合；C. JSON 配置文件 | 选 C，与 CI 友好；CLI 仍可覆盖配置文件 |
| 0.2.0 | 配置文件读取容错 BOM | PowerShell `Out-File -Encoding utf8` 默认写 BOM，`JSON.parse` 不接受 | A. 文档要求用户去 BOM；B. 读取时统一剥离 | 选 B，在 Windows 主流工作流下零摩擦 |
| 0.2.0 | 交互流程加摘要确认页 | 原流程一回车就开始写盘，错填无撤销窗口 | A. 加 `--force` 提示；B. 加摘要确认 | 选 B，低成本高收益护栏 |
| 0.2.0 | 删除非规范命名的学习心得 | 同一人两份近似文件违反"主事实源唯一"原则 | A. 合并为一份；B. 保留命名规范一致的版本 | 选 B，与 `知识库/00_知识库总目录.md` 登记一致 |
| 0.2.0 | 知识库补 PO / 质量 / 学习路径三份指南 | 原模板偏 Scrum 流程，缺产品价值、质量门禁、个人成长 | A. 扩展现有 06 文件；B. 拆为独立三份 | 选 B，便于按角色单独引用 |
| 0.2.0 | 新增"模板演进与反向回流指南" | 模板要可持续演进，必须把"团队 Retro → 模板更新"的路径定下来 | A. 在 README 加一段；B. 独立指南 | 选 B，并自此约定每次 L3 改动追加 CHANGELOG |

> 历史教训：0.2.0 前半段（P0/P1）改动先于反向回流指南落地，违反了自己定的规则。这是把"先做再立规则"作为模板首次回流案例记录下来的原因。
