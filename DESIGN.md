# create-scrum-team-workspace 设计说明

## 目标

从零生成一个可用于 Scrum 团队协同开发的项目工作区，让团队启动时同时拥有：

- 项目运行层：产品发现、Backlog、Sprint、风险、会议、度量和知识库。
- 工程设计层：ADR、API、数据模型、前端设计系统、技术债和 Spike。
- 代码落地层：仓库清单、按策略生成的轻量代码骨架、`TeamWork/`、分支和 Git 身份。
- 角色协同层：角色表、角色卡、能力矩阵、备份机制、Sprint 0 分工、SM 作战手册和可配置命名系统。

## 项目背景类型与仓库策略

项目背景类型说明项目从何而来，通常保持稳定：

| type | 适用场景 | 生成后的第一动作 |
| --- | --- | --- |
| `new` | 从零新项目 | 填愿景、建首批 Backlog、确认工程技术栈 |
| `legacy` | 存量项目重构 | 做现状诊断、冻结参考源、确定迁移边界 |
| `product` | 成熟产品迭代 | 梳理 Backlog、容量规划、补齐质量门禁 |
| `prototype` | 原型转正 | 复核技术选型、重建目录、补齐安全和测试基线 |

仓库策略说明本 Sprint 如何处理代码，每次 Planning 可重新确认：

| 策略 | 含义 | 是否生成目标仓 |
| --- | --- | :---: |
| `reuse` | 继续使用现有远端仓库 | 否 |
| `import` | 整理已有/散落代码后导入新仓 | 是 |
| `rewrite` | 旧仓生产维护，新仓验证并渐进切换 | 是 |
| `create` | 从零建立新代码仓 | 是 |

默认映射仅用于减少输入：`new -> create`、`legacy -> rewrite`、
`product -> reuse`、`prototype -> import`。用户可以覆盖，且 Sprint 中可变更。

## 角色系统

固定角色槽位保持 Scrum 与工程职责稳定：

| 槽位 | 默认职责 |
| --- | --- |
| `po` | 产品负责人 / PM |
| `sm` | Scrum Master |
| `tl` | TL / Sr.BE |
| `midbe` | Mid.BE / QA API |
| `srfe` | Sr.FE / UX |
| `midfe` | Mid.FE / QA E2E |
| `fs` | Full-stack / DevOps |

命名系统只改变“显示名称”和工具链占位，不改变职责槽位。这样既能保留团队氛围，也能保证文档、分支、目录和 Git 身份可追溯。

## CLI 能力

```bash
node index.mjs <project-name> \
  --type=new|legacy|product|prototype \
  --repo-strategy=reuse|import|rewrite|create \
  --source-repo=<url-or-path> \
  --preset=tech|myth|wuxia|compass|studio|greek \
  --repo=<repo-name> \
  --role.midfe=Aurora \
  --email.po=jobs@acme.com \
  --config=./scrum.config.json \
  --interactive \
  --dry-run \
  --git-root=repo|workspace|none \
  --worktrees \
  --role-test-commits \
  --remote=<url> \
  --push \
  --sprint=1 \
  --no-git
```

### 交互流程

`--interactive`（或缺省项目名时自动触发）会逐项询问项目、仓库和角色信息。角色套装选定后先展示全量角色表，用户可按槽位或全部调整名称与邮箱；随后确认 worktree、角色测试提交、远端和推送策略，最后在**摘要确认**页输入 Y 才会写盘。

### 占位符约定

- 文件内容统一使用 `{{KEY}}` 占位符（如 `{{PROJECT_NAME}}`、`{{REPO_NAME}}`、`{{CREATED_DATE}}`）。
- 文件名/目录名使用 `__KEY__` 形式（如 `__PROJECT_NAME__.code-workspace`、`10_代码仓库/__REPO_NAME__/`），避免与文档正文里的普通词产生冲突。

## 取舍

- 保持零依赖，便于复制、审阅和本地执行。
- 不直接生成具体技术栈代码，避免把模板绑定到某个框架。
- 项目工作区文档是主事实源，代码仓库不再生成重复的 `01-docs` 文档中心；代码仓库只保留 `apps/`、`infra/`、README、可执行资产和链接。
- 生成文档多于生成业务代码，因为 Scrum 团队协同的最大风险通常是输入输出、角色边界、证据链和集成节奏不清。
- 默认使用 `feature/sprint-<n>/<topic>-<name>-<role>` 分支名，目录名允许显示名称但会清理明显危险字符。
- 默认只建立文档治理仓；代码仓是 Sprint 0 下游决策，显式 `--git-root=repo` 才在创建时立即建立。
- 远端配置和推送是两个动作，避免提供 URL 后意外写入远端。
- 角色就绪提交是可选验证证据，只存在于各自分支，不自动合入主线。

## 安全边界与适用范围

签核系统提供的是**受信任团队的流程完整性与审计追踪**，而非操作者身份认证：

- 命令级 Git 身份只解决配置竞争，**不提供身份认证**；任何有仓库写权限的人
  都能运行 `sign --role=<他人>`，工具会主动用该角色的姓名/邮箱提交并通过作者校验。
  因此当前保证的是“工件与声明身份一致”，不是“操作者就是本人”。
- Notice 一致性摘要是可重算的防漂移校验和，**非秘密、非身份认证、非阅读证明**。
- 适用：受信任团队的流程完整性与审计追踪；高保证环境仍需平台账号、保护分支、
  签名提交或审批认证（见 ADR “签名提交高保证选配”）。
