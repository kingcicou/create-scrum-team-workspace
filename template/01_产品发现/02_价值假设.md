# 价值假设

> 本表是产品发现阶段最关键的"为什么做"。每一条价值假设都要可证伪、可度量。
> 写法与方法论详见 `知识库/Scrum/07_产品发现与价值排序指南.md` §2。

## 1. 假设清单

| 编号 | 用户/角色 | 场景 | 能力/改变 | 期望结果（可度量） | 验证信号/指标 | 状态 | 验证时间 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| H1 |  |  |  |  |  | 🔵 待验证 |  |
| H2 |  |  |  |  |  | 🔵 待验证 |  |

**状态图例**：🔵 待验证 · 🟢 验证中 · ✅ 成立 · ❌ 不成立 · ⏸️ 已退场

## 2. 假设书写模板

```text
我们认为，对 <用户/角色>
在 <场景> 下
提供 <能力/改变>
能够带来 <可度量的结果>
我们将通过 <信号/指标> 验证它是否成立。
```

## 3. 与 Backlog 的关系

- 每个价值假设至少绑定一个 Epic（写入 `02_产品待办/01_Epics/`）
- Sprint Review 用本表的"验证信号"对照实际增量，更新状态列
- 已验证不成立的假设应**显式退场**（状态改 ❌ + 退场理由），不要静默删除

## 4. 反模式

- 把"功能清单"当假设：缺"为什么"
- 假设无法证伪：例如"提升用户体验"
- 假设无验证指标：只能凭感觉判断成立
- 已不成立的假设仍占 Backlog 容量
