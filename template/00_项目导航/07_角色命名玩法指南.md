# 角色命名玩法指南

角色名称是协作体验的一部分，但不能牺牲工程可追溯性。本模板把角色拆成三层：

| 层级 | 用途 | 建议 |
| --- | --- | --- |
| 显示名称 | 文档、看板、沟通氛围 | 可以有一点风格，例如技术大神、神话、罗盘、工作室 |
| 角色职责 | Scrum 帽子和工程边界 | 保持稳定，例如 PO、SM、TL/Sr.BE、Mid.BE、Sr.FE、Mid.FE、FS |
| 工具链名称 | 目录、分支、邮箱占位 | 使用英文、拼音或短代号；避免空格、括号、点号和特殊符号 |

## CLI 示例

使用默认世界技术大神套装：

```bash
node index.mjs my-product
```

选择一套命名风格：

```bash
node index.mjs my-product --preset=compass
```

只调整个别角色：

```bash
node index.mjs my-product --preset=tech --role.midfe=Aurora --role.fs=Atlas
```

交互式逐个确认：

```bash
node index.mjs --interactive
```

查看全部预设：

```bash
node index.mjs --list-presets
```

## 命名建议

- `PO` 和 `SM` 可以偏人格化，帮助团队记住“价值”和“流程”两种守护职责。
- 编码角色建议保留工程意味，例如 `Forge`、`Kernel`、`Canvas`、`Bridge`。
- 如果使用中文显示名称，分支名会自动退回到角色代号；为避免重复，建议额外约定英文短代号。
- 角色命名可以有趣，但不要让它替代真实责任人。真实姓名、账号、邮箱仍应在工具权限清单中记录。

## 不建议

- 不用 `Evan(Mid.FE_QA)` 作为目录名。括号和点号会增加脚本转义成本。
- 不用中文分支名。Git 支持，但 CI、URL、Shell 和跨平台脚本更容易出问题。
- 不把角色名当成固定头衔。成员变动时，应更新 `roles.config.json`、角色文档和 Git 身份配置。
