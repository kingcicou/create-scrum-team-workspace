# 团队协作交互协议与 SM 播报模板

> 本文件是成员与 SM 日常协作的唯一操作入口。它用于减少追问和重复写文档，不是新增审批流程。

## 1. 使用边界

- 事实只在责任源更新一次：Backlog 管优先级，输入输出总表管交付物，监控台管当前快照，Git/CI 管代码证据。
- 群聊消息只引用任务 ID、责任人和证据链接，不另造一份事实台账。
- 没有状态变化时不重复上报；Daily Scrum 只同步变化、依赖、风险和下一步。
- SM 不代替成员确认专业事实，也不把推测写成确定结论。
- 普通成员只需使用本文件；`知识库/Scrum/12_SM流程监控与角色行动决策规范.md` 是 SM/流程教练选读材料。

## 2. 团队共同语言

任务状态只使用以下六类：

| 状态 | 含义 |
|---|---|
| 未开始 | 输入已知，但尚未投入执行 |
| 进行中 | 正在产出可交付结果 |
| 等待输入 | 自身可继续准备，但关键输入尚未到位 |
| 阻塞 | 无法继续，必须由明确责任人解除 |
| 待评审 | 产出已提交，等待评审、验收或合并 |
| 已完成 | 已满足 DoD，并有可核验的完成证据 |

成员应在以下事件发生时主动同步，而不是等待 SM 逐人追问：

1. 开始任务或状态发生变化。
2. 新增等待、阻塞、风险或范围变化。
3. 产出进入评审、验收或交接。
4. 任务完成，或原定完成时间明显失真。
5. 发现 SM 公布的信息与事实不一致。

## 3. 成员向 SM 上报

### 3.1 状态包

```text
@SM [<任务ID>][<状态>]
正在：<当前活动>
等待/阻塞：无 | @<角色> <所需输入>，期望 <时间>
将输出：<成果>，预计 <时间>
交接：完成后由 @<角色> 继续 <后续活动>
SM同步：待确认 | 已同步 <时间>
证据：<Story/PR/CI/文档链接>
```

没有等待或阻塞时写“无”，不要为了填满模板制造内容。

### 3.2 SM 确认

```text
已同步 [<任务ID>]：<状态>
关键依赖：无 | <前置任务/输入>
完成后解锁：无 | @<角色> <后续活动>
下次检查：<事件或时间>
```

SM 无法核实时，应回复“待澄清”并点名缺失事实，不能静默地自行推断。成员收到“已同步”后，无须再向其他台账重复抄写同一事实。

### 3.3 状态纠偏

```text
@SM 状态纠偏 [<任务ID>]
SM记录：<当前公开状态>
实际情况：<正确状态>
影响：<对依赖、时间或范围的影响>
证据：<链接>
```

SM 应修正责任源或监控快照，并在原播报渠道明确回复“已纠偏”。纠偏是正常的信息校准，不作为成员过失。

## 4. SM 群聊快报

适用于站会后、关键状态变化或管理者询问。目标是一屏内可直接复制到群聊。

```text
【Sprint <编号>｜<日期时间>｜<绿/黄/红>】
目标：<Sprint Goal>
当前：<生命周期阶段> / <Sprint阶段>；下一门禁：<门禁>

关键链：
@<角色A> <任务> -> @<角色B> <任务> -> <评审/门禁> -> Sprint Goal

并行工作：
- @<角色C> <任务>（输出：<成果>）
- @<角色D> <任务>（输出：<成果>）

角色行动：
- @<角色A> [<状态>] 现在 <活动>；下一步 <活动>；预计 <时间>
- @<角色B> [<状态>] 等待 @<角色A> <输入>；可先行 <活动>

阻塞/风险：
- <问题>｜责任人 @<角色>｜解除条件 <条件>｜检查时间 <时间>

本轮需确认：
- @<角色> 请在 <时间> 前确认 <事项>
```

没有阻塞时写“无”。没有证据支撑的状态标为“待确认”，不要用绿色掩盖未知。

## 5. SM 流程全景

适用于 Sprint 中期校准、Review 前或复盘。它回答“从 Sprint 开始以来发生了什么”，不要求每次查询都生成。

```text
# Sprint <编号> 流程全景（截至 <时间>）

## 目标与可信度
- Sprint Goal：<目标>
- 总体状态：<绿/黄/红及原因>
- 信息新鲜度：<最新更新时间；待确认角色>

## 时间线与因果
1. <时间>：@<角色> 完成 <活动/成果>，解锁 @<角色> <后续活动>。
2. <时间>：@<角色> 与 @<角色> 并行开展 <活动>，在 <门禁> 汇合。
3. <时间>：<事件> 导致 <影响>，由 @<角色> 处理。

## 当前关键链
<前置任务> -> <当前任务> -> <后续任务> -> <评审/发布>

## 角色全景
| 角色 | 已完成 | 当前状态/活动 | 等待或阻塞 | 下一输出/交接 |
|---|---|---|---|---|
| @<角色> | <成果> | <状态与活动> | <输入/无> | <成果 -> 后续角色> |

## 下一汇合门
- 门禁：<名称>
- 必须到齐：<输入/证据>
- 尚缺：<缺口及责任人>
- 决策时间：<时间>
```

时间线只记录改变流程走向的事件：完成、交接、阻塞、解锁、范围或决策变化。不要把每条聊天记录改写成项目史。

## 6. 单角色状态卡

当用户只问“某个角色现在做什么”时，SM 使用本模板，不展开整份全景：

```text
@<角色> [<状态>] <任务ID>
当前：<活动>
等待/阻塞：<输入、责任人、解除条件>
下一输出：<成果与预计时间>
完成后：交接 @<角色>，解锁 <活动>
SM掌握：已确认 <时间> | 待本人确认
证据：<链接>
```

## 7. 无法判断时

```text
当前不能可靠判断 <角色/任务> 的状态。
已知：<最后确认事实及时间>
缺少：<状态、输入、证据或责任人>
请 @<角色> 按“状态包”补充；补齐前不安排依赖方正式开工。
可先行：<不依赖缺失输入的准备工作>
```

## 8. 最小协作能力

| 对象 | 必须掌握 |
|---|---|
| 所有成员 | 用统一状态表达事实；说清等待、输出和交接；发现偏差及时纠正 |
| SM/流程教练 | 聚合责任源；区分时序、依赖和并行；把事实翻译成可执行播报；显式标记未知 |
| PO/TL/FS | 对优先级、技术依赖、集成门禁等职责内事实及时确认，不让 SM 代替决策 |

判断这套协议是否有效，只看三件事：成员能否立即知道自己要做什么，依赖方能否知道何时接手，未知信息能否被快速点名和补齐。
