# SM 教练查询与回复模板

## 1. 使用原则

本模板用于回答“当前处于什么阶段、谁先做、哪些可并行、某角色现在和随后做什么”。

- 先讲关键因果链，再讲角色任务。
- 先列当前并行泳道，再列下一汇合门。
- 区分事实、规则、建议和待确认信息。
- 没有可靠事实时停止推断，不为了给答案而派活。
- 依赖和状态引用 ID/链接，不复制第二套 Backlog。

## 2. 标准提问

```text
教练，请读取当前 Sprint 流程监控台、输入输出总表、Backlog、风险台账和
Git/CI 证据，按时间线和因果关系回答：
1. 当前主阶段和各工作流状态；
2. 关键路径上谁先做、谁后做；
3. 当前有哪些互不依赖的并行活动；
4. 下一汇合门缺少什么；
5. <角色> 现在做什么、完成后做什么、等待什么、会解锁谁。
信息不足或过期时请明确指出，不要推断。
```

## 4. 精简回复

适合站会或即时消息：

```text
快照：<时间>，<Sprint/Day>，可信度 <状态>
当前阶段：<主阶段>
关键链：A -> B -> G1 -> 目标
现在并行：①角色/活动 ②角色/活动 ③角色/活动
下一门禁：G1；已具备 <...>；尚缺 <...>；决策人 <...>

<角色>：
- 现在：...
- 随后：完成 ... 后做 ...
- 等待：...（负责人/预计时间）
- 可并行：...
- 解锁：...
- 暂停边界：...

依据：Story / 总表 / 风险 / PR / CI
```

## 5. 无法判断时

```text
当前状态无法可靠判断。

缺失或冲突：
- <事实源>：最后更新时间 / 冲突内容

请由 <责任人> 在 <时间> 前更新。更新前可以安全开展的仅有：
- <低耦合、可逆活动>

暂不建议：
- <依赖未决输入的不可逆实现>
```

## 6. 回复质量检查

- [ ] 先给因果链，再给角色行动。
- [ ] 明确哪些活动可并行及其原因。
- [ ] 每个等待项都有输入负责人和预计时间。
- [ ] 每项行动说明完成后解锁什么。
- [ ] 汇合门列出已具备、尚缺和决策人。
- [ ] 没有把建议写成未经确认的指令。
- [ ] 引用了可追溯的 Story、总表、风险或代码证据。
