# 团队协作交互协议与 SM 播报模板

> 本文件是成员与 SM 日常协作的唯一操作入口。它用于减少追问和重复写文档，不是新增审批流程。

## 0. 先查什么、选哪个模板

SM 回答前只查四类事实：项目首页确认当前 Sprint，任务表确认 Task/依赖，风险表
确认等待或阻塞，Git/CI/产物确认完成证据。事实不足就使用“无法判断”，不补写推测。

| 对方的问题或场景 | 使用 |
|---|---|
| “我该怎么向 SM 汇报？” | §3.1 状态包 |
| “当前 Sprint 整体到哪了？” | §4 群聊快报 |
| “从 Sprint 开始以来发生了什么？” | §5 流程全景 |
| “某个角色现在做什么？” | §6 单角色状态卡 |
| 事实源过期、成员尚未确认 | §7 无法判断 |
| Review/Retro 谁已追加、是否覆盖他人 | §9 评审完整性检查 |
| 需要发起、纠偏或关闭角色签核 | §10 签核通知 |

### 0.1 提问示例

```text
请按群聊快报说明当前 Sprint 的关键链、并行任务和需要确认的人。
请只回答 <成员> 当前任务、等待输入、下一输出，以及 SM 是否已确认。
请给出本 Sprint 从 Planning 至今的流程全景，只记录解锁、阻塞和决策变化。
```

### 0.2 完整示例

提问：“FS 现在做什么？”

```text
@<FS成员> [等待输入] TASK-05
当前：准备 CI 变更影响评估，尚未修改流水线。
等待/阻塞：等待 @TL 确认新技术栈构建命令；其余环境检查可先行。
下一输出：CI 变更方案、验证结果和运行证据。
完成后：交接 @TL 评审，解锁集成门禁。
SM掌握：已确认 <时间>
证据：<任务表/commit/CI链接>
```

## 1. 使用边界

- 事实只在责任源更新一次：Backlog 管优先级，输入输出总表管交付物，监控台管当前快照，Git/CI 管代码证据。
- 群聊消息只引用任务 ID、责任人和证据链接，不另造一份事实台账。
- 普通进度只更新 Sprint 任务行；跨角色等待、阻塞、交接或事实纠偏时才发送状态包。
- 没有状态变化时不重复上报；Daily Scrum 只同步变化、依赖、风险和下一步。
- SM 不代替成员确认专业事实，也不把推测写成确定结论。
- 普通成员只需使用本文件；`知识库/Scrum/12_SM流程监控与角色行动决策规范.md` 是 SM/流程教练选读材料。

## 2. 团队共同语言

任务状态只使用以下六类：

| 状态 | 含义 |
|---|---|
| 未开始 | 输入已知，但尚未投入执行 |
| 进行中 | 正在产出可交付结果 |
| 等待输入 | 自身可继续准备，但关键输入尚未到位 |
| 阻塞 | 无法继续，必须由明确责任人解除 |
| 待评审 | 产出已提交，等待评审、验收或合并 |
| 已完成 | 已满足 DoD，并有可核验的完成证据 |

成员应在以下跨角色事件发生时主动同步，而不是等待 SM 逐人追问：

1. 新增等待、阻塞、风险或范围变化。
2. 产出进入跨角色评审、验收或交接。
3. 原定完成时间明显失真并影响依赖方。
4. 发现 SM 公布的信息与事实不一致。

任务开始、普通进行中和完成但未影响他人时，只更新任务表，不额外发消息。

## 3. 成员向 SM 上报

### 3.1 状态包

```text
@SM [<任务ID>][<状态>]
正在：<当前活动>
等待/阻塞：无 | @<角色> <所需输入>，期望 <时间>
将输出：<成果>，预计 <时间>
交接：完成后由 @<角色> 继续 <后续活动>
SM同步：待确认 | 已同步 <时间>
证据：<Story/PR/CI/文档链接>
```

没有等待或阻塞时写“无”，不要为了填满模板制造内容。

### 3.2 SM 确认

```text
已同步 [<任务ID>]：<状态>
关键依赖：无 | <前置任务/输入>
完成后解锁：无 | @<角色> <后续活动>
下次检查：<事件或时间>
```

SM 无法核实时，应回复“待澄清”并点名缺失事实，不能静默地自行推断。成员收到“已同步”后，无须再向其他台账重复抄写同一事实。

### 3.3 状态纠偏

```text
@SM 状态纠偏 [<任务ID>]
SM记录：<当前公开状态>
实际情况：<正确状态>
影响：<对依赖、时间或范围的影响>
证据：<链接>
```

SM 应修正责任源或监控快照，并在原播报渠道明确回复“已纠偏”。纠偏是正常的信息校准，不作为成员过失。

## 4. SM 群聊快报

适用于站会后、关键状态变化或管理者询问。目标是一屏内可直接复制到群聊。

```text
【Sprint <编号>｜<日期时间>｜<绿/黄/红>】
目标：<Sprint Goal>
当前：<生命周期阶段> / <Sprint阶段>；下一门禁：<门禁>

关键链：
@<角色A> <任务> -> @<角色B> <任务> -> <评审/门禁> -> Sprint Goal

并行工作：
- @<角色C> <任务>（输出：<成果>）
- @<角色D> <任务>（输出：<成果>）

角色行动：
- @<角色A> [<状态>] 现在 <活动>；下一步 <活动>；预计 <时间>
- @<角色B> [<状态>] 等待 @<角色A> <输入>；可先行 <活动>

阻塞/风险：
- <问题>｜责任人 @<角色>｜解除条件 <条件>｜检查时间 <时间>

本轮需确认：
- @<角色> 请在 <时间> 前确认 <事项>
```

没有阻塞时写“无”。没有证据支撑的状态标为“待确认”，不要用绿色掩盖未知。

## 5. SM 流程全景

适用于 Sprint 中期校准、Review 前或复盘。它回答“从 Sprint 开始以来发生了什么”，不要求每次查询都生成。

```text
# Sprint <编号> 流程全景（截至 <时间>）

## 目标与可信度
- Sprint Goal：<目标>
- 总体状态：<绿/黄/红及原因>
- 信息新鲜度：<最新更新时间；待确认角色>

## 时间线与因果
1. <时间>：@<角色> 完成 <活动/成果>，解锁 @<角色> <后续活动>。
2. <时间>：@<角色> 与 @<角色> 并行开展 <活动>，在 <门禁> 汇合。
3. <时间>：<事件> 导致 <影响>，由 @<角色> 处理。

## 当前关键链
<前置任务> -> <当前任务> -> <后续任务> -> <评审/发布>

## 角色全景
| 角色 | 已完成 | 当前状态/活动 | 等待或阻塞 | 下一输出/交接 |
|---|---|---|---|---|
| @<角色> | <成果> | <状态与活动> | <输入/无> | <成果 -> 后续角色> |

## 下一汇合门
- 门禁：<名称>
- 必须到齐：<输入/证据>
- 尚缺：<缺口及责任人>
- 决策时间：<时间>
```

时间线只记录改变流程走向的事件：完成、交接、阻塞、解锁、范围或决策变化。不要把每条聊天记录改写成项目史。

## 6. 单角色状态卡

当用户只问“某个角色现在做什么”时，SM 使用本模板，不展开整份全景：

```text
@<角色> [<状态>] <任务ID>
当前：<活动>
等待/阻塞：<输入、责任人、解除条件>
下一输出：<成果与预计时间>
完成后：交接 @<角色>，解锁 <活动>
SM掌握：已确认 <时间> | 待本人确认
证据：<链接>
```

## 7. 无法判断时

```text
当前不能可靠判断 <角色/任务> 的状态。
已知：<最后确认事实及时间>
缺少：<状态、输入、证据或责任人>
请 @<角色> 按“状态包”补充；补齐前不安排依赖方正式开工。
可先行：<不依赖缺失输入的准备工作>
```

## 8. 最小协作能力

| 对象 | 必须掌握 |
|---|---|
| 所有成员 | 用统一状态表达事实；说清等待、输出和交接；发现偏差及时纠正 |
| SM/流程教练 | 聚合责任源；区分时序、依赖和并行；把事实翻译成可执行播报；显式标记未知 |
| PO/TL/FS | 对优先级、技术依赖、集成门禁等职责内事实及时确认，不让 SM 代替决策 |

判断这套协议是否有效，只看三件事：成员能否立即知道自己要做什么，依赖方能否知道何时接手，未知信息能否被快速点名和补齐。

## 9. Review/Retro 评审完整性检查

共享文件追加前后均执行：

```bash
node tools/review-status.mjs 03_迭代运行/<Sprint目录>/<Review或Retro文件>.md
git diff -- 03_迭代运行/<Sprint目录>/<Review或Retro文件>.md
```

工具列出“评审意见追加”下的唯一三级标题，并在重复 section 或重复标题时返回失败。
Reviewer 汇报“本人已完成”即可，可附当前观察名单；**完整名单和缺席确认由 SM
负责**，不要求每位 Reviewer 重复维护一份名单。

## 10. 签核通知

签核的**编排 Owner 永远是 SM**；成员只阅读受影响内容并追加本人事件。SM 不得把
“汇总范围、发通知、核验闭环”转交给任一被签核成员。

### 10.1 入队首签通知

```text
【角色手册首签｜<Campaign ID>｜基线 <Vx>】
目的：确认每个人知道自己的职责、边界、输入输出和周期动作。
覆盖范围：<Change ID>
签核范围：角色卡、责任表、周期任务清单、§4 本人事件。
签核人员：@<全体成员名单>
你要做：
1. 从项目首页进入角色手册，阅读自己的角色卡和必读最小集；
2. 找到当前任务，跑通一次真实任务或完成 SM 指定的上手演练；
3. 在 §4 追加唯一 Event ID：Campaign、目标基线、覆盖 Change ID、时间、证据方式 auto。
截止：<时间>
完成回复：使用 §10.5 成员签核回复。
疑问：直接 @SM；不要转交他人代签。
```

### 10.2 变更重签通知

```text
【角色手册重签｜<Campaign ID>｜<旧基线> → <新基线>】
变更：<文件/章节及一句话影响>
覆盖：<CHG-ID 列表>
本次范围：@<受影响成员>；未列名成员无需动作。
阅读：<精确章节链接>
动作：确认影响 → 在 §4 追加本人事件 → 使用 §10.5 回复。
截止：<时间>
编排/验收：@SM
```

### 10.3 误派纠偏通知

```text
【签核任务纠偏｜<Campaign ID>｜基线 <Vx>】
原通知问题：<误派/漏派/范围错误>。
正确范围：@<应签成员>。
无需动作：@<被误派成员>，此前任务撤回，不影响其现有签核。
依据：角色手册 §5.2 与本轮签核任务卡。
下一步：@SM 更新范围并在完成后发布闭环结果。
```

### 10.4 完成闭环通知

```text
【签核闭环｜<Campaign ID>｜基线 <Vx>】
范围：<首签/重签>；应签 <n> 人，完成 <n> 人。
结果：<成员 → Event ID → Git commit>。
核验：角色、目标基线、Change ID 覆盖和 Git 作者均正确；审计无待处理项。
结论：本轮签核关闭。后续仅在入队或职责/边界/IO 实质变化时再触发。
```

### 10.5 成员签核回复

```text
【签核完成｜<Campaign ID>】
角色/成员：<角色> / <姓名>
Event ID：<全局唯一，如 EVT-20260703-FS-001>
从基线 → 目标：<Vx> → <Vy>
本次覆盖：<CHG-xxx, CHG-yyy | BASELINE-Vy>
说明：<incremental / catch-up / full-rebaseline>
提交：已由本人提交；请 SM 运行审计器自动反查 commit 与作者。
```

跨版本补签必须列出实际覆盖的 Change ID。签到 V1.4 不自动等于曾在 V1.2
发布时间签过；它只表示本次事件覆盖了 V1.2 的相关变化。

### 10.6 全量重基线通知

```text
【全量重基线｜<Campaign ID>｜目标 <Vx>】
原因：<历史不可验证 / 模型迁移 / 重大角色重构>。
范围：@全员；覆盖：BASELINE-<Vx>。
阅读：当前角色卡、责任表、周期任务清单和重大变化摘要；不要求重读所有旧版本。
动作：每人追加一个 full-rebaseline 事件，并按 §10.5 回复。
效果：恢复当前 Vx 有效性；旧版本历史缺口继续保留，不倒填为已签。
```
