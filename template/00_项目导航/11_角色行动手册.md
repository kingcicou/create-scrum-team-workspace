# 角色行动手册

> **一页纸找到"我的职责/边界/输入输出/行动/必读/当前任务/周期职责"。**
> 本手册是**聚合入口**：绝大多数内容用链接指向权威源，不复制。
> 新成员上手不必通读全部规范——**读你自己那张卡 + 跑通一次真实任务**即可。
>
> 📍 **入口关系**：团队唯一总入口是 [项目首页](00_项目首页.md)；本手册是从首页「30 分钟上手」进入的**角色专属页**。
> 角色显示名见 [02_角色与联系方式](02_角色与联系方式.md) / `roles.config.json`；下表以**角色帽子**为键。
>
> 🧢 **小团队一人多帽**：卡按帽子不变，兼任者读自己所有帽子的卡并逐帽签核；适配步骤见
> [知识库/项目模板/05_小团队角色裁剪指南](../知识库/项目模板/05_小团队角色裁剪指南.md) §5/§6。

## 使用方式

1. 找到**你的角色卡**（§1）。
2. 按卡里的"必读最小集"读你需要的规范（不是全部）。
3. 从项目首页确认当前是 Planning、执行还是已关闭，再进入相应事实源。
4. 收到 SM 的 Campaign ID 后，运行通知中的
   Notice 中本人对应的完整 `sign` 命令（含 `--notice=<sha256>`）。工具从
   `roles.config.json` 取得命令级提交身份并创建本人 Event；不要修改仓库 Git 身份。
5. 只有本人职责/边界/IO 实质变化时才重签；签核不是 Sprint 开工门禁。

> **实时签核有效性**（谁因规范变更需重签）不看 §4 静态表，而看生成器输出的
> 签核状态由本地生成的 `文档索引/06_停滞审计.md` 提供；首次查看先按 [文档索引/README.md](文档索引/README.md) 生成。它自动计算 Change ID 覆盖并反查 Git 证据。

## 2. 治理责任表（谁生成谁治理）

**两条原则**：① 内容级——**谁生成谁治理**，文档 owner 负责自己的元数据/状态/证据/三要素；
② 聚合级——**SM 触发+分发不代治**，SM 跑脚本、读报告、点名、记度量，绝不替 owner 补内容。

| 治理/周期任务 | 触发 | 处理人 | 原则 |
|---|---|---|---|
| 单文档 Frontmatter / 状态 / 证据 / 三要素 | 产出即维护 | 该文档 owner | 谁生成谁治理 |
| 跑 `generate_doc_index.py` + 读报告 | Sprint 末/异常触发 | SM | SM 触发不代治 |
| managed 文档缺失/非法字段 | SM 点名后 | 各文档 owner | 历史/exempt 不追溯 |
| 协作完整性（评审输入缺失/误改正文） | 出现即修 | 对应 Reviewer / 文档 Owner | 谁的输入谁补，正文由 Owner 汇总 |
| 治理健康度记入度量快照 | 每 Sprint 末 | SM | SM 管度量 |
| 签核变更/批次 + 事件跟踪 | 入队或职责/边界/IO 变化 | SM 用 signoff CLI 建批次；成员生成本人事件文件 | 命令级身份；共享目录写入串行化 |
| 门禁 blocked 登记风险表 | 出现即登 | 负责该门禁的角色 | 谁生成谁治理 |

## 5. 签核基线与变更日志

| Change ID | 手册基线 | 日期 | 变更内容 | 影响角色 |
|---|:--:|---|---|---|
| CHG-100 | V1.5 | {{CREATED_DATE}} | 首版；含关闭同步、变化触发式 CI、签核编排、纠偏批次与独立事件文件模型 | ALL |

### 5.1 签核批次

| Campaign ID | 模式 | 目标基线 | 覆盖范围 | 应签角色 | 发起时间 | 截止时间 | 状态 | 关闭证据 |
|---|---|---|---|---|---|---|---|---|
| 由 bootstrap 生成 | initial | V1.5 | 按全局审计逐角色生成 | ALL | 创建时/建库后 | 默认 +72h advisory | 待创建 | 创建者运行一次 `signoff bootstrap` |

### 5.2 SM 变更→重签 SOP（触发与通知）

**谁触发**：改动某"必读规范"的 owner **有义务通知 SM**；SM 也可 `git log` 主动发现 version bump。

**SM 五步**：
1. **感知**：收到 owner 通知，或 git log 发现必读规范 version bump。
2. **判定**：实质变更（影响职责/边界/IO/流程）→ 需重签；纯措辞 → 不需。
3. **更新手册**：§5 登记唯一 Change ID 和影响角色，升内容基线并运行索引。
4. **建批次/通知**：先确认除生成型文档索引外工作区干净，再运行
   `signoff.mjs prepare --from-audit --actor=sm --due="<未来时间>"` 自动生成
   逐角色范围；只有 `verify` 为 `OK` 且 `exact=yes` 时才运行
   `publish --actor=sm`，并原样转发 NOTICE-BEGIN/END 全文。
5. **核验旧证据**：legacy commit 必须进入当前 `HEAD` 历史；只存在于某人本地
   对象库的悬空提交按无效处理。
5. **闭环**：`status` 检查本批次；仅 SM 执行 `close --actor=sm`。工具刷新项目
   全局审计，待处理非零时拒绝关闭。

**重签通知模板**：

```text
【重签通知】手册基线 V1.0 → V1.1
- 变更规范：<规范名> <旧版> → <新版>
- 改了什么：<一句话>
- 影响角色：<全员 / TL+编码角色 / ...>
- 你要做：① 读变更章节<链接>；② 运行通知中的 `signoff.mjs sign` 命令
- 截止：<日期>。生成器 [06_停滞审计·签核状态] 显示未覆盖 Change ID。
```

### 5.3 签核编排协议

| 动作 | 唯一责任人 | 不可转移的责任 |
|---|---|---|
| 首签准备 | 项目创建者 | 确认角色姓名/邮箱与 workspace Git；运行一次 `bootstrap --actor=sm` 并推送；只有创建者恰好兼任 FS 时才由 FS 执行 |
| 建立后续基线 | SM | 登记 Change 后运行索引；`prepare --from-audit --actor=sm` 自动取范围 |
| 发起通知 | 首签由工具；后续由 SM | 原样转发已提交 Notice，不手工压缩、补写或合并角色 |
| 执行签核 | 各成员本人 | 运行 sign；不改 Git 配置；工具自动排队等待写锁 |
| 结果核验 | SM | 运行 status；检查覆盖、不可变文件和命令级署名 |
| 宣布闭环 | SM | `close --actor=sm`；Campaign 局部与项目全局均归零才成功 |

**铁律：SM 可以请成员澄清内容，但不能把签核编排任务转给其他成员。**
被转交者应回复“签核编排属于 SM；我只处理本人签核”。

五种场景：

1. **入队首签**：项目创建者（不是默认的 FS 职责）在角色确认和事实源建库后运行一次 `bootstrap`；
   工具建立 initial Campaign 和不可变 Notice，SM 原样转发并跟踪，成员本人运行
   sign。阅读范围固定为角色卡、责任表、周期任务清单和签核规则。首签不是代码
   开工门禁，但必须作为入队闭环跟踪；详细命令见 09 §11.1。
2. **变更重签**：SM 只通知 `resign-roles` 中受职责、边界或 IO 实质影响的角色；
   未列名成员不得因版本号变化被要求重签。
3. **仅 SM 自签**：若范围只有 SM，SM 不派任务，直接自查、运行 sign、重跑
   审计并发布闭环结果。
4. **跨版本补签**：一个 catch-up 事件可以覆盖多个非连续 Change ID，但只表示
   在本次补签中确认；没有原始证据时不得倒推为曾在旧版本发布时签过。
5. **关闭后纠偏**：已关闭批次后来发现漏签、代签或无效证据时，不改写旧批次；
   SM 建立 `corrective` Campaign；需要理解新规则时可用 `BASELINE-Vx` 覆盖全员。

签核前，SM 必须把以下字段写进 Campaign；`publish` 据此生成不可变 Notice：

```text
Campaign ID：<SIGN-ID>｜目标基线：<Vx>｜模式：initial / incremental / catch-up / full-rebaseline / corrective
覆盖范围：<Change ID 或 BASELINE-Vx>
应签人员：<明确名单>
阅读范围：<精确章节>
截止时间：<时间>
验收：本人 Event ID + 覆盖范围 + 命令级 Git 作者；全局审计无待处理
```

## 6. 分阶段团队与分批签核

不是"先组建完整团队再做 Sprint 0"，而是"先任命足以开展发现与决策的**核心启动团队**，
由他们通过 Sprint 0 定技术、仓库和后续**交付团队**形态"。

### 6.1 核心启动团队 ≠ 交付团队

`roles.config.json` 每角色有 `status`：`active`（当前激活，需签核）/ `optional`（涉及时参与）
/ `planned`（Sprint 0 明确需求后再激活）。`bootstrap`、启动通知与审计**只读 active 角色**。

| 角色 | Sprint 0 职责（最低核心要求） | 默认 core 档 |
|---|---|:--:|
| PO | 明确产品目标、价值、范围与优先级 | active |
| SM | 组织 Planning、暴露依赖、维护流程事实 | active |
| TL/资深技术 | 现状诊断、技术选项、模块边界、风险与 Spike | active |
| Sr.FE/UX | 涉及体验或前端产品时参与 | optional |
| Mid.BE/Mid.FE/FS | 模块与任务清晰后再加入 | planned |

> FS 只有在仓库迁移、环境约束或交付平台影响技术决策时才需提前加入。

### 6.2 分批签核模式

| 模式 | 对象 | 触发 |
|---|---|---|
| `initial-core` | 当前 active 核心团队 | 项目创立（`bootstrap`） |
| `onboarding` | 新加入/新激活成员 | 入队或 planned→active |
| `role-change` | 职责/帽子/边界变化的成员 | 帽子或 IO 改变 |
| `governance-change` | 受影响成员 | 核心规范实质升级（登记新 CHG） |

**"当前 Sprint 0 核心团队全部完成首签"才能启动 Sprint 0 派工**（kickoff 前提）。

### 6.3 后续成员入队（RC3 手动步骤）

1. 编辑 `roles.config.json`，把该角色 `status` 由 `planned` 改为 `active`（填真实姓名/邮箱）。
2. SM 发起增量首签，只要求新成员签当前基线，**不要求核心团队重签**：
   ```bash
   node tools/signoff.mjs prepare --mode=onboarding --roles=<角色> --due=+72h --actor=sm
   node tools/signoff.mjs publish --campaign=<ID> --actor=sm
   ```
3. 新成员首签有效前可阅读与准备，但不应领取正式实施任务。

> 现已支持 `team.mjs add` / `team.mjs assign`：
> - `add` 用于成员入队（建议随后由 SM 发起 onboarding 批次）；
> - `assign` 用于成员帽子变更（建议随后由 SM 发起 role-change 批次）；
> - 写入前后会自动校验团队模型，且不会按姓名/邮箱自动合并成员身份。

### 6.4 仓库审批 ≠ 人员签核

代码仓创建是**技术与投资决定**（PO/TL 审批），走 `propose → approve → check → apply`
（见 `03_迭代运行/Sprint-0-启动/仓库决策卡.md`），决策工件在 `.team/repo-decisions/`，
与 `.team/signoffs/` 的人员签核**分属不同工件**——"签了规范"不等于"批准这次技术投资"。
