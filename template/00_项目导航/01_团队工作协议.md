# 团队工作协议

## 迭代节奏

- Sprint 长度：2 周（可调整）。
- Daily Scrum：每日固定时间，15 分钟内。
- Refinement：Sprint 中至少 1 次。
- Review + Retro：Sprint 末举行。

## 协作规则

- 需求不满足 DoR 不进入 Sprint。
- 完成必须满足 DoD，不以“代码写完”为准。
- 阻塞超过半天必须显性暴露。
- 兼任工作必须进入 Backlog 并占容量。
- 重要决策写入 ADR 或 `90_会议与决策/决策记录.md`。

## 状态同步

- 成员在任务开始、状态变化、等待/阻塞、交接或完成时主动同步，不等待 SM 逐人追问。
- 团队统一使用“未开始、进行中、等待输入、阻塞、待评审、已完成”六类状态。
- 成员用状态包说明当前活动、等待、下一输出、交接对象和证据；SM 回复“已同步”或“待澄清”。
- 发现 SM 播报与事实不一致时，成员立即发起状态纠偏；纠偏不作为过失。
- 具体格式统一使用 `00_项目导航/09_SM教练查询与回复模板.md`，不另建日报或重复台账。

## 代码规则

- 不直接在 `main` 上开发。
- FS 维护 `sprint-x` 集成分支。
- 每个编码角色使用独立 worktree/clone。
- 分支名使用英文小写 hyphen。
- 每个工作区配置本地 Git `user.name` 和 `user.email`。

## DoR

- Story 有明确价值或风险说明。
- AC 可验证。
- 依赖和风险明确。
- 团队完成估算。
- 能在一个 Sprint 内完成，或已拆分。

## DoD

- AC 全部满足。
- PR Review 通过。
- CI 或本地等价验证通过。
- 测试证据已记录。
- PO 验收通过。
- Story 工作区记录分支、PR、测试、验收和集成结论。
