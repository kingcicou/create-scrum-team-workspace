# 团队工作协议

## 迭代节奏

- Sprint 长度：2 周（可调整）。
- Daily Scrum：每日固定时间，15 分钟内。
- Refinement：Sprint 中至少 1 次。
- Review + Retro：Sprint 末举行。

## 协作规则

- 需求不满足 DoR 不进入 Sprint。
- 完成必须满足 DoD，不以“代码写完”为准。
- 阻塞超过半天必须显性暴露。
- 兼任工作必须进入 Backlog 并占容量。
- 重要决策写入 ADR 或 `90_会议与决策/决策记录.md`。
- 只在启动、首次开工、合并、关闭四个门禁做快速确认；正常通过不另填表。
- 普通事实写回任务行、PR/MR、CI 或 Review；偏离计划、职责、证据契约或目标时才登记例外并裁决。

## 分工与任务

- PO、TL、资深工程师负责目标、边界、模块拆分、复杂度和验收/Review。
- 复杂度 `L` 或跨模块任务由资深成员承担或结对；`S/M` 优先交给中初级成员。
- 执行成员默认只维护
  `03_迭代运行/<当前Sprint>/01_Sprint任务表与流程看板.md` §4 中的状态和证据，
  不另写日报或实现说明。
- SM 维护节奏、依赖和清障，不替专业角色拆技术任务，也不代写交付物。
- 任务预计超过 2 天时，由拆分责任人继续拆分。

## 状态同步

- 普通进度只更新当前 Sprint 任务表 §4；等待/阻塞、跨角色交接或纠偏时主动通知 SM。
- 团队统一使用“未开始、进行中、等待输入、阻塞、待评审、已完成”六类状态。
- 成员用状态包说明当前活动、等待、下一输出、交接对象和证据；SM 回复“已同步”或“待澄清”。
- 发现 SM 播报与事实不一致时，成员立即发起状态纠偏；纠偏不作为过失。
- 具体格式统一使用 `00_项目导航/09_SM教练查询与回复模板.md`，不另建日报或重复台账。

## 外部环境

- Git 平台、Docker、云环境或专有依赖暂不可用，但仍能编码、测试或写清操作步骤时，不记为开发阻塞。
- 状态标记为“待指定环境验证”，由 FS 指定验证人、目标环境和最晚时间。
- 验证人回报环境/版本、依赖、命令、预期、实际结果、证据和偏差处理。
- 只有它阻断实现、Sprint Goal、集成、发布或安全门禁时才升级为“阻塞”。
- waiver 允许继续推进或关闭时间盒，但不能把未满足的 Goal 改写为已完成。

## 代码规则

- 不直接在 `main` 上开发。
- FS 维护 `sprint-x` 集成分支。
- 每个编码角色使用独立 worktree/clone。
- 分支名使用英文小写 hyphen。
- 每个工作区配置本地 Git `user.name` 和 `user.email`。
- 首次开工运行 `node tools/code-preflight.mjs`；失败时先修正，不提交后再追查。
- Reviewer 给终态结论，FS/平台规则负责合并；平台不可用时留下五项等价评审证据。

## DoR

- Story 有明确价值或风险说明。
- AC 可验证。
- 依赖和风险明确。
- 团队完成估算。
- 能在一个 Sprint 内完成，或已拆分。

## DoD

- AC 全部满足。
- PR Review 通过。
- CI 或本地等价验证通过。
- 测试证据已记录。
- PO 验收通过。
- Sprint 任务表已记录证据；复杂、跨角色 Story 才按需建立独立工作区。
