# 输入输出管理规范

**适用范围：** {{PROJECT_NAME}} 全体成员  
**生效日期：** {{CREATED_DATE}}  
**核心规则：** 谁产出、谁更新。普通 Task 更新 Sprint 任务表；长期交付事实才更新总表。

> 默认采用轻量治理：普通任务只更新 Sprint 任务表；只有长期产物和关键决策
> 进入本规范与 06 总表。高冲突文档才按需启用 `知识库/Scrum/13_文档协作与并发控制规范.md`。

---

## 1. 目录分区原则

| 区域类型 | 目录 | 用途 | 操作规则 |
| --- | --- | --- | --- |
| 项目工作区 | `00_项目导航/` ~ `07_度量改进/`、`90_会议与决策/` | 团队协作文档、评估报告、Backlog、度量、会议决策 | 在此产出和更新 |
| 代码仓库区 | {{REPO_WORKSPACE_LOCATION}} | 可执行代码、测试脚本、迁移脚本、CI/CD 配置、构建脚本 | 通过 Git 管理；仓库角色登记在 `10_代码仓库/00_仓库清单.md` |
| 本地协同区 | 实际代码仓的 `TeamWork/` | 本机 worktree/clone 工作目录 | 已加入代码仓 `.gitignore`，禁止提交 |
| 参考源 | 存量项目代码、历史文档、竞品资料 | 只读分析输入 | 不直接写入结论和新产物 |
| 知识库 | `知识库/` | 方法论、模板、复盘经验 | 低频维护，经评审后更新 |

## 2. 主事实源边界

| 信息类型 | 主事实源 | 代码仓库是否重复维护 |
| --- | --- | --- |
| 产品愿景、用户场景、路线图 | `01_产品发现/` | 不重复，只在 README 链接 |
| Epic、Story、AC、需求变更 | `02_产品待办/` | 不重复，只在 PR/提交信息引用 |
| Sprint 计划、风险、评审、回顾 | `03_迭代运行/` | 不重复，只保留代码证据链接 |
| 架构、ADR、API、数据模型、技术债 | `04_工程设计/` | 不重复；可执行契约/迁移脚本进入代码仓库 |
| 测试策略、用例、缺陷、质量报告 | `05_质量验证/` | 测试脚本在代码仓库，质量结论在工作区 |
| 发布策略、部署、回滚、Runbook | `06_发布运维/` | 部署脚本在代码仓库，说明和决策在工作区 |
| 度量、健康检查、改进项 | `07_度量改进/` | 不重复 |
| PR、CI、构建产物 | Git 平台 / CI 平台 | Sprint 任务表记录链接；长期质量/发布结论才进总表 |

> 原则：文档主事实源在项目工作区；代码仓库承载可执行资产。不要在代码仓库里再维护一套 `01-docs` 文档中心。

## 3. 输入输出行为准则

| # | 规则 | 说明 |
| --- | --- | --- |
| 1 | 规范先于产出 | 规范中引用的目录必须已经存在，否则成员会把产物写到错误位置 |
| 2 | 完成即更新 | 普通 Task 更新当前 Sprint 任务表；长期产物再更新 `06_团队输入输出总表/00_索引.md` |
| 3 | 路径自填 | 产出人以真实存放路径为准，不预填不存在路径 |
| 4 | 链接优于复制 | 历史资料、PR、CI、设计稿、看板项用链接引用 |
| 5 | Draft 不入永久目录 | 必须共享的 Draft/WIP 放入 Sprint 的 `评估产物/<角色>/`；个人临时稿不入库 |
| 6 | 可执行制品进仓库 | CI、Docker、迁移、测试脚本、配置样例进入仓库清单指定的当前开发仓 |
| 7 | 描述和决策进工作区 | 架构取舍、发布说明、质量结论、Runbook 进入对应文档目录 |
| 8 | 知识库低频维护 | 执行产物不得直接写入知识库；实践稳定后由 SM 发起沉淀 |

### 3.1 新建文档门

普通实现任务默认不新建 Markdown。满足以下任一条件才新建或扩充正式文档：

1. 存在需要长期追溯的取舍、风险或后果，例如 ADR。
2. 产物是两个以上角色共同消费的契约，例如 API、数据或体验约束。
3. 操作会重复执行且容易出错，例如发布、回滚或故障 Runbook。
4. 法务、安全、验收或审计要求独立证据。

否则，更新 Sprint 任务表的状态与证据链接即可。会议纪要、个人日报、重复
周报和“为了说明做过什么”的实现文档默认不创建。

### 3.2 角色文档负担

| 工作类型 | 主责 | 默认产出 |
| --- | --- | --- |
| 价值/范围决策 | PO | Goal、Backlog、AC、必要决策记录 |
| 架构/模块拆分 | TL、资深工程师 | 技术全景、必要 ADR/契约、任务拆分 |
| 普通实现/测试 | 中初级工程师 | 代码、测试、任务状态、证据链接 |
| 流程协调 | SM | 依赖/阻塞更新；不代写专业文档 |
| 环境/部署 | FS | 自动化资产；外部条件不足时提交最小操作说明 |

## 5. 中间产物与正式产物

| 产物类型 | 定义 | 存放规则 |
| --- | --- | --- |
| Draft / WIP | 尚未评审通过的草稿、调研、拆分方案 | `03_迭代运行/<Sprint>/评估产物/<角色>/` |
| 正式文档 | 已评审或可被他人消费的材料 | 对应长期目录，如 `04_工程设计/`、`05_质量验证/` |
| 可执行制品 | 代码、脚本、配置、迁移、自动化测试 | 仓库清单指定的当前开发仓 |
| 描述文档 | 说明、结论、Runbook、发布策略 | 项目工作区对应目录 |

迁移流程：

```text
Draft/WIP -> 评审通过 -> 提炼为正式文档 -> 写入长期目录 -> 更新总表
                         -> 可执行部分进入代码仓库 -> README/Story 链接
```


| 领域缩写 | 含义 |
| --- | --- |

## 7. 总表更新指南

| 场景 | 操作 |
| --- | --- |
| 新建产出 | 新增总表行，填写 ID、名称、位置、主责、状态、依赖、评审人 |
| 修改已有产出 | 更新变更摘要、文档版本、评审状态 |
| 交付给下一角色 | 消费者填写领取人，产出人填写交付时间 |
| 评审通过 | 状态改为已归档或已通过，记录评审人 |
| 阻塞 | 状态改为阻塞，备注写清影响、负责人和期望解决时间 |
| 放入代码仓库 | 总表记录仓库路径、PR/CI 链接或 README 入口 |
| 影响当前 Sprint 流动 | 总表更新后，在 `03_迭代运行/<Sprint>/01_Sprint任务表与流程看板.md` 引用 ID，更新任务依赖和门禁 |

> 总表记录跨 Sprint 的“交付事实”；Sprint 任务表与流程看板记录本轮 Task 和流动。
> 两者通过 ID/链接关联，不复制完整任务内容。

## 8. 文档评审机制

| 文档类型 | 必须评审人 | 可选评审人 |
| --- | --- | --- |
| 产品文档、Backlog | PO | TL、SM、QA |
| 架构、ADR、API、数据模型 | TL / Sr.BE | FE、BE、FS |
| 质量策略、测试策略 | QA 帽子角色 + TL | 受影响开发成员 |
| 发布、运维、Runbook | FS / DevOps + TL | PO |
| UX/UI、设计系统 | Sr.FE / UX + PO | Mid.FE |
| 管理流程文档 | SM + PO | 全员 |

评审只针对达到“新建文档门”的正式产物。普通任务状态、测试结果和操作说明
由任务 Reviewer 验证，不启动独立文档评审。

评审状态：

```text
待评审 -> 评审中 -> 已通过
                   -> 需修订 -> 再次提交
```

多人并行评审时，不直接在目标文档正文追加意见。产出人/Owner 先建立
`_review-inputs/<目标文档ID或短名>/<member-id>.md`，Reviewer 只写自己的评审输入；
Owner 汇总采纳结论后再更新正文、06 总表和任务证据。Review/Retro 纪要同样适用：
成员写个人输入文件，SM 维护主纪要。

`_review-inputs/` 是正式评审证据，必须提交 Git；个人未定稿草稿放
`_review-drafts/` 或 `Temp/review-drafts/`，由 `.gitignore` 忽略，不作为事实源。

## 9. 违规处理

| 违规 | 处理 |
| --- | --- |
| 产物写入参考源或知识库 | 迁移到正确目录，并在站会说明 |
| Task 状态变化超过 1 个工作日未更新 | SM 提醒；反复发生才进入 Retro |
| 代码仓库和工作区双写同一文档 | 确认主事实源，删除或改为链接 |
| `TeamWork/` 被提交 | 立即移除提交内容，补充 `.gitignore` 检查 |
| Draft 被当正式文档消费 | 移回 Sprint 暂存区，补评审状态 |

## 10. Sprint 0 补救清单模板

| 违规/缺口 | 现状 | 目标 | 责任人 | 截止时间 | 状态 |
| --- | --- | --- | --- | --- | --- |
| 产物位置混乱 | 待检查 | 迁移至正确目录 | SM + 产出人 | Sprint 0 | 🔵 |
| 总表未同步 | 待检查 | 产出和总表一致 | SM | Sprint 0 | 🔵 |
| 目录不存在 | 待检查 | 规范引用目录全部存在 | 各领域负责人 | Sprint 0 | 🔵 |
| 代码/文档混放 | 待检查 | 可执行制品和说明文档分离 | FS | Sprint 0 | 🔵 |

---

> 本规范来自 QFD_Ark Sprint 0 的实际教训：规范要先于产出，总表必须同步，成熟项目不能套从零模板，目录必须立即可用，代码和文档要分清事实源。

## 11. 按冲突风险升级协作控制

不要给所有文件套同一套流程。按实际冲突风险选择：

| 级别 | 场景 | 控制 |
| --- | --- | --- |
| L0 默认 | 任务状态、个人执行、低冲突说明 | Owner 直接更新，Reviewer 在任务表确认 |
| L1 关键产物 | ADR、契约、发布/回滚、跨角色设计 | 单一 Owner + 1-2 名 Reviewer |
| L2 高冲突 | 多人频繁并发修改同一正式文档 | 再启用分段 Owner、PR、CODEOWNERS 或平台审批 |

GitHub/GitLab/Codeup 自动评审是可选加速器，不是团队开工前置条件。没有平台
能力时，可由 Owner 提交文件、Reviewer 在任务表或评审段确认。完整升级策略见
`知识库/Scrum/13_文档协作与并发控制规范.md`。

## 12. Frontmatter 元数据 Schema（可检索地基）

给需要长期治理的正式文档统一机器可读元数据，`tools/generate_doc_index.py` 就能按
**角色 / 迭代 / 领域 / 阶段 / 状态** 生成可点击索引，解决“产出散落、按维度找不到”。

达到“新建文档门”（§3.1）且需要长期治理的产物标记
`governance: managed`；历史、入口、骨架和未纳管材料默认 `exempt`，仍可查询，
但不形成追溯治理债。managed 文档头部 YAML 必须包含：

| 字段 | 必填 | 取值 / 格式 | 说明 |
| --- | --- | --- | --- |
| `id` | 是 | 对应 06 总表行 ID | 追溯锚点 |
| `title` | 是 | 文档标题 | 与 H1 一致 |
| `owner` | 是 | 单一负责人（显示名） | 唯一改写权归属 |
| `reviewers` | 否 | 列表 | 无评审可省 |
| `domain` | 是 | `PM`/`PO`/`BE`/`FE`/`QA`/`OPS`/`UX` | 领域维度 |
| `phase` | 是 | `需求`/`概要设计`/`详细设计`/`编码`/`测试`/`部署运维`/`治理` | 阶段维度 |
| `sprint` | 是 | `Sprint-0` / `跨迭代` 等 | 时间线维度 |
| `type` | 是 | `评估报告`/`规范`/`ADR`/`API契约`/`数据模型`/`测试用例`/`Runbook`/`度量`/`会议纪要`/`决策` | 产出物类型 |
| `status` | 是 | `draft`/`review`/`approved`/`locked` | 与总表评审状态对齐 |
| `version` | 是 | 如 `V1.0` | 版本 |
| `last-updated` | 是 | `YYYY-MM-DD` | 更新日期 |
| `governance` | 是 | `managed` / `exempt` | 是否进入治理健康度和缺字段审计 |

`domain`/`phase`/`type` 只能取上表枚举值。L0、历史和关闭 Sprint 不追溯补录；
生成器只报告 managed 文档的缺失或非法字段。

> 生成命令：`python tools/generate_doc_index.py`，结果写入 `00_项目导航/文档索引/`。

## 13. 产出契约与任务卡（执行前概览 + 可审计）

### 13.1 产出形态唯一

同一任务只允许一种被认定为“交付物”的形态，落点唯一：

| 形态 | 适用 | 落点 |
| --- | --- | --- |
| a 正式文档 | 长期决策/跨角色契约/审计证据 | 对应永久目录（带 §12 Frontmatter） |
| b 代码+测试+任务表状态 | 普通实现任务 | 代码仓 + Sprint 任务表证据 |
| c 任务表内联结论 | 轻量结论 | Sprint 任务表“输出/证据”列 |

群聊/口头不是交付物；对他人文档的意见写入
`_review-inputs/<目标文档>/<member-id>.md`，禁止改他人正文
（多人评审见 `知识库/Scrum/13` §8，提交署名见 `知识库/Scrum/11` §8）。

### 13.2 任务卡（开工前概览）

达到“新建文档门”或有跨角色依赖/门禁的任务，开工前填一张轻量卡（写在 Sprint 任务表里，不新建文件）：

```text
### 🃏 [任务ID] · [标题]
- Owner / 级别：[名] · [D/A/I/V/O]
- 输入：[依赖 + 状态要求，如 C01(approved)]；无则“—”
- 输出：形态 [a/b/c] · 落点 [唯一路径] · type=[类型] · [需/免]评审([评审人])
- 门禁：🟢 放行 / 🔴 阻塞([原因])
- 时序：先行 / 并行([对象]) / 延后([对象])
- DoD/证据：[完成判据 + 证据链接]
```

资深/领域负责人在拆分时创建；执行人开工前消费、执行中只回填状态与证据，不擅改契约。

### 13.3 非终态的证据三要素（可追溯/可审计）

`等待输入 / 阻塞 / 待评审` 是非终态，必须带三要素，缺任一 = 审计缺陷：

| 要素 | 例 |
| --- | --- |
| 原因 | 缺任务卡 / 依赖未定版 / 等 ACK |
| 责任对象 | @谁 |
| 起始日期 | since 日期 |

写法：`等待输入：缺任务卡 @资深X since 日期`。任务卡缺失时，执行人不硬等也不擅自开工，
而是置此状态让“谁忘了”可见、可归因。门禁标 blocked 必须在 `03_迭代运行/<Sprint>/03_风险与障碍.md` 有登记。

> 门禁不拆，给它装可审计的证据链。可选检测由 `tools/generate_doc_index.py` 的“停滞审计”提供。
