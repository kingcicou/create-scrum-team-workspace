# 输入输出管理规范

**版本：** V1.0  
**适用范围：** {{PROJECT_NAME}} 全体成员  
**生效日期：** {{CREATED_DATE}}  
**核心规则：** 谁产出、谁更新、谁埋尾巴。产出不填总表 = 没完成。

---

## 1. 目录分区原则

| 区域类型 | 目录 | 用途 | 操作规则 |
| --- | --- | --- | --- |
| 项目工作区 | `00_项目导航/` ~ `07_度量改进/`、`90_会议与决策/` | 团队协作文档、评估报告、Backlog、度量、会议决策 | 在此产出和更新 |
| 代码仓库区 | `10_代码仓库/{{REPO_NAME}}/` | 可执行代码、测试脚本、迁移脚本、CI/CD 配置、构建脚本 | 通过 Git 管理，README 链回项目工作区 |
| 本地协同区 | `10_代码仓库/{{REPO_NAME}}/TeamWork/` | 本机 worktree/clone 工作目录 | 已加入 `.gitignore`，禁止提交 |
| 参考源 | 存量项目代码、历史文档、竞品资料 | 只读分析输入 | 不直接写入结论和新产物 |
| 知识库 | `知识库/` | 方法论、模板、复盘经验 | 低频维护，经评审后更新 |

## 2. 主事实源边界

| 信息类型 | 主事实源 | 代码仓库是否重复维护 |
| --- | --- | --- |
| 产品愿景、用户场景、路线图 | `01_产品发现/` | 不重复，只在 README 链接 |
| Epic、Story、AC、需求变更 | `02_产品待办/` | 不重复，只在 PR/提交信息引用 |
| Sprint 计划、风险、评审、回顾 | `03_迭代运行/` | 不重复，只保留代码证据链接 |
| 架构、ADR、API、数据模型、技术债 | `04_工程设计/` | 不重复；可执行契约/迁移脚本进入代码仓库 |
| 测试策略、用例、缺陷、质量报告 | `05_质量验证/` | 测试脚本在代码仓库，质量结论在工作区 |
| 发布策略、部署、回滚、Runbook | `06_发布运维/` | 部署脚本在代码仓库，说明和决策在工作区 |
| 度量、健康检查、改进项 | `07_度量改进/` | 不重复 |
| PR、CI、构建产物 | Git 平台 / CI 平台 | Story 工作区和总表记录链接与结论 |

> 原则：文档主事实源在项目工作区；代码仓库承载可执行资产。不要在代码仓库里再维护一套 `01-docs` 文档中心。

## 3. 输入输出行为准则

| # | 规则 | 说明 |
| --- | --- | --- |
| 1 | 规范先于产出 | 规范中引用的目录必须已经存在，否则成员会把产物写到错误位置 |
| 2 | 存前填表 | 放入文件或提交可执行制品时，同步更新 `06_团队输入输出总表.md` |
| 3 | 路径自填 | 产出人以真实存放路径为准，不预填不存在路径 |
| 4 | 链接优于复制 | 历史资料、PR、CI、设计稿、看板项用链接引用 |
| 5 | Draft 不入永久目录 | Draft/WIP 放入 Sprint 目录的 `评估产物/<角色>/` 或 Story 工作区 |
| 6 | 可执行制品进仓库 | CI、Docker、迁移、测试脚本、配置样例进入 `10_代码仓库/{{REPO_NAME}}/` |
| 7 | 描述和决策进工作区 | 架构取舍、发布说明、质量结论、Runbook 进入对应文档目录 |
| 8 | 知识库低频维护 | 执行产物不得直接写入知识库；实践稳定后由 SM 发起沉淀 |

## 4. 角色产出边界

{{ROLE_TABLE}}

| 角色帽子 | 主要产出位置 | 禁止事项 |
| --- | --- | --- |
| PO / PM | `01_产品发现/`、`02_产品待办/` | 不把口头需求直接塞入开发分支 |
| SM | `00_项目导航/`、`03_迭代运行/`、`07_度量改进/` | 不把总表当事后补账 |
| TL / Sr.BE | `04_工程设计/`、关键 PR Review | 不让架构决策只停留在群聊 |
| Mid.BE / QA API | `05_质量验证/`、后端测试脚本、缺陷记录 | 不把“本地跑通”当质量结论 |
| Sr.FE / UX | `04_工程设计/04_前端设计系统/` | 不只还原界面而忽略体验和可访问性 |
| Mid.FE / QA E2E | 前端组件、E2E、快照、联调证据 | 不提交未自测页面 |
| FS / DevOps | `10_代码仓库/`、`06_发布运维/` | 不做隐形 DevOps 工作，不让 CI 红灯过夜 |

## 5. 中间产物与正式产物

| 产物类型 | 定义 | 存放规则 |
| --- | --- | --- |
| Draft / WIP | 尚未评审通过的草稿、调研、拆分方案 | `03_迭代运行/<Sprint>/评估产物/<角色>/` |
| 正式文档 | 已评审或可被他人消费的材料 | 对应长期目录，如 `04_工程设计/`、`05_质量验证/` |
| 可执行制品 | 代码、脚本、配置、迁移、自动化测试 | `10_代码仓库/{{REPO_NAME}}/` |
| 描述文档 | 说明、结论、Runbook、发布策略 | 项目工作区对应目录 |

迁移流程：

```text
Draft/WIP -> 评审通过 -> 提炼为正式文档 -> 写入长期目录 -> 更新总表
                         -> 可执行部分进入代码仓库 -> README/Story 链接
```

## 6. 文件命名约定

统一模板：

```text
[领域缩写]_[内容摘要]_[角色名].md
```

| 领域缩写 | 含义 |
| --- | --- |
| `MGT` | 管理、流程、SM |
| `PO` | 产品、市场、路线图 |
| `BE` | 后端、API、数据 |
| `FE` | 前端、组件、页面 |
| `UX` | 体验、交互、可访问性 |
| `QA` | 质量、测试、缺陷 |
| `OPS` | 运维、CI/CD、发布 |

示例：

```text
BE_API契约评审_Compass.md
UX_交互走查_Horizon.md
OPS_CI_CD基线_Voyager.md
```

## 7. 总表更新指南

| 场景 | 操作 |
| --- | --- |
| 新建产出 | 新增总表行，填写 ID、名称、位置、主责、状态、依赖、评审人 |
| 修改已有产出 | 更新变更摘要、文档版本、评审状态 |
| 交付给下一角色 | 消费者填写领取人，产出人填写交付时间 |
| 评审通过 | 状态改为已归档或已通过，记录评审人 |
| 阻塞 | 状态改为阻塞，备注写清影响、负责人和期望解决时间 |
| 放入代码仓库 | 总表记录仓库路径、PR/CI 链接或 README 入口 |
| 影响当前 Sprint 流动 | 总表更新后，在 `03_迭代运行/<Sprint>/01_Sprint流程监控台.md` 引用 ID，更新阶段门禁和角色行动 |

> 总表记录“交付事实”，Sprint 流程监控台记录“这些事实对当前流动意味着什么”。
> 两者通过 ID/链接关联，不复制完整任务内容。

## 8. 文档评审机制

| 文档类型 | 必须评审人 | 可选评审人 |
| --- | --- | --- |
| 产品文档、Backlog | PO | TL、SM、QA |
| 架构、ADR、API、数据模型 | TL / Sr.BE | FE、BE、FS |
| 质量策略、测试策略 | QA 帽子角色 + TL | 受影响开发成员 |
| 发布、运维、Runbook | FS / DevOps + TL | PO |
| UX/UI、设计系统 | Sr.FE / UX + PO | Mid.FE |
| 管理流程文档 | SM + PO | 全员 |

评审状态：

```text
待评审 -> 评审中 -> 已通过
                   -> 需修订 -> 再次提交
```

## 9. 违规处理

| 违规 | 处理 |
| --- | --- |
| 产物写入参考源或知识库 | 迁移到正确目录，并在站会说明 |
| 产出超过 4h 未更新总表 | SM 提醒，超过 1 天进入 Retro |
| 代码仓库和工作区双写同一文档 | 确认主事实源，删除或改为链接 |
| `TeamWork/` 被提交 | 立即移除提交内容，补充 `.gitignore` 检查 |
| Draft 被当正式文档消费 | 移回 Sprint 暂存区，补评审状态 |

## 10. Sprint 0 补救清单模板

| 违规/缺口 | 现状 | 目标 | 责任人 | 截止时间 | 状态 |
| --- | --- | --- | --- | --- | --- |
| 产物位置混乱 | 待检查 | 迁移至正确目录 | SM + 产出人 | Sprint 0 | 🔵 |
| 总表未同步 | 待检查 | 产出和总表一致 | SM | Sprint 0 | 🔵 |
| 目录不存在 | 待检查 | 规范引用目录全部存在 | 各领域负责人 | Sprint 0 | 🔵 |
| 代码/文档混放 | 待检查 | 可执行制品和说明文档分离 | FS | Sprint 0 | 🔵 |

---

> 本规范来自 QFD_Ark Sprint 0 的实际教训：规范要先于产出，总表必须同步，成熟项目不能套从零模板，目录必须立即可用，代码和文档要分清事实源。
