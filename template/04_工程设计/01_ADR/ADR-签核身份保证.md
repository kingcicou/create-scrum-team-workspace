# ADR 签核身份保证与签名提交高保证路线

## 状态

approved（记录路线；**不进入默认签核流程**）

## 背景

`tools/signoff.mjs` 用命令级 Git 身份（`git -c user.name/user.email`）+ 事件文件
单次提交 + 首次作者/邮箱校验，保证「工件与声明身份一致」。

但这**不是操作者身份认证**：任何拥有仓库写权限的人都能运行
`sign --role=<他人>`，工具会主动用该角色的姓名/邮箱提交并通过作者校验。因此
当前保证的是流程完整性与审计追踪，不是「操作者就是本人」。

## 决策

将「签名提交（GPG/SSH signing）」作为**高保证模式的可选路线**记入演进，
不进入默认流程。默认流程继续面向受信任团队。

## 高保证模式设计（未来选配）

1. 角色映射到 GPG/SSH 公钥指纹（`roles.config.json` 增加 `signingKey`）。
2. Event 首次提交必须带有效签名（`git commit -S`）。
3. 事件证据校验增加 `git verify-commit`。
4. 校验签名指纹与角色映射一致。
5. 支持密钥轮换、吊销与历史指纹保留。

## 边界与限制

- 签名证明的是**私钥持有**，不必然证明本人意图。
- 若多人或共享环境能访问同一私钥，认证能力退化。
- 因此私钥必须置于**独立账号、独立工作区或硬件密钥**中。

## 适用范围

- 默认（受信任团队）：命令级身份 + 审计追踪即可。
- 高保证（跨组织、合规、外部审计）：叠加平台账号、保护分支、签名提交或审批认证。

## 后果

- 正面：给出清晰的身份保证升级路径，不过度工程化默认流程。
- 负面：高保证模式需要密钥治理，落地成本较高，故延后。
