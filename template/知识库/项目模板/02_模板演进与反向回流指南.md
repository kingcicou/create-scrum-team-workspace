# 模板演进与反向回流指南

> 本模板由 `create-scrum-team-workspace` 生成。团队改进了它，不应该只留在本项目里。这份指南定义"团队 Retro → 模板演进"的回流路径。

## 1. 谁可以发起回流

任何成员都可以提出，由 SM 把关入口；最常见的回流来源：

| 来源 | 触发时刻 |
| --- | --- |
| Retro 改进项 | 改进项被验证有效、且对其他团队也成立 |
| Sprint 0 复盘 | 项目类型化策略需要新增/裁剪 |
| 缺陷复盘 | 模板缺一项门禁、清单、目录 |
| 跨项目对比 | 多个项目踩了同一个坑 |
| 工具/平台变化 | CI、看板、协作工具升级 |

## 2. 回流的三个层级

| 层级 | 改什么 | 落地位置 |
| --- | --- | --- |
| L1 项目内 | 仅当前项目适用 | 项目工作区，不回流 |
| L2 知识库 | 经验、原则、反模式可被复用 | 项目知识库 → 模板 `template/知识库/` |
| L3 生成器 | 结构、占位符、CLI、默认值要变 | `index.mjs` + `template/` 目录结构 |

判断口诀：**只影响这次 = L1；影响下一个团队 = L2；影响"怎么生成" = L3**。

## 3. 回流流程

```text
1. 提案：Retro 决议或 SM 收集，写一条"模板演进提案"
2. 评估：在本项目验证一个 Sprint，确认稳定有效
3. 抽象：剥离项目特有信息，提炼为通用原则/检查项
4. 提交：向 create-scrum-team-workspace 仓库提 PR
5. 合并后：在团队总表登记一条"模板已演进"
6. 验收：核对来源、L2知识、L3操作、测试和版本记录，不以“模板已改”代替完整回流
```

提案有两种形式，按范围选用：

**轻量提案**（写入本项目 `90_会议与决策/决策记录.md` 即可）：

```markdown
## 模板演进提案 · YYYY-MM-DD

- 起因：
- 现象：
- 在本项目的验证：
- 建议层级：L1 / L2 / L3
- 建议落点：
- 风险/兼容性：
- 提议人 / 评审人：
```

**迭代计划**（多项变更、有实施项和验收点时使用）——范本见 `项目模板/03_角色工作区自动化迭代计划.md`，推荐结构：

```markdown
# <迭代主题>

**迭代主题：** ...
**决策日期：** YYYY-MM-DD
**适用版本：** x.y.z

## 1. 目标
## 2. 需求与采纳结论（表格）
## 3. 实施项（P0/P1 + 验收标准）
## 4. 风险控制
## 5. 验证场景
## 6. Definition of Done
```

## 4. L2 知识库回流的最小检查

回流到模板知识库的内容必须满足：

- [ ] 是经验/原则/清单/反模式，不是当期数据。
- [ ] 删除了项目名、人员真名、客户名等敏感信息。
- [ ] 提供"反例"或"修正"，不是单纯吐槽。
- [ ] 与现有知识库无重复；如重叠则采用合并修改而非新增。
- [ ] 在 `99_来源索引.md` 中登记原始项目来源。
- [ ] 若经验改变了角色动作或默认行为，已同步对应 L3 操作模板；没有只写原则。

## 5. L3 生成器回流的最小检查

修改 `index.mjs` 或 `template/` 时：

- [ ] 占位符遵循约定：内容用 `{{KEY}}`、路径名用 `__KEY__`。
- [ ] 新增 CLI 选项同时更新 `printHelp`、`README.md`、`DESIGN.md`。
- [ ] 模板新增目录必须真实可用（至少一个 README 或说明）。
- [ ] 用 `--dry-run` 验证清单变化符合预期。
- [ ] 跑一次完整生成，确认无未替换占位符。
- [ ] 为新增/修改的能力补一条 `test/` 测试，`npm test` 全绿。
- [ ] 已判断并完成对应 L2 知识提炼；没有用 L3 功能修复替代经验沉淀。

### 5.1 完整回流 Definition of Done

一项跨项目经验只有同时满足适用层级的证据才算完成：

| 层 | 必要证据 |
|---|---|
| 来源 | 项目复盘/事故/Retro 可定位，`99_来源索引.md` 已登记 |
| L2 知识 | 通用原则、反模式或清单已合并到现有知识文档 |
| L3 操作 | 行为受影响时，角色手册、SOP、检查表或工具已更新 |
| 验证 | 行为变更有测试；纯知识变更至少有生成内容断言 |
| 发布 | CHANGELOG、DESIGN 和版本标签可追溯 |
| 项目闭环 | 项目侧回流总账记录最终落点与状态 |

只完成 L1 项目修复、只完成 L2 文档或只完成 L3 模板代码，都不能写“已完整回流”。

## 6. 不应回流到模板的内容

| 内容 | 理由 | 应去哪 |
| --- | --- | --- |
| 当期 Backlog、Story、AC | 是项目执行物 | 留在项目工作区 |
| 客户/业务专有名词 | 不通用 | 留在项目工作区或自建词典 |
| 一次性手工脚本 | 不稳定 | 项目代码仓库的 `infra/` |
| 个人风格的注释/段子 | 干扰其他团队 | 个人学习心得 |
| 实验中的 Spike 结论 | 未验证 | 等稳定后再回流 |

## 7. 模板版本与变更日志

模板自身在 `create-scrum-team-workspace` 仓库中维护 `CHANGELOG`；每次合并 L3 回流后追加一条。生成的项目工作区不需要复制 CHANGELOG，但可在 `00_项目导航/00_项目首页.md` 记录"基于模板版本 X.Y"。

## 8. 反模式

| 反模式 | 后果 | 修正 |
| --- | --- | --- |
| Retro 改进项直接改模板 | 模板被未经验证的经验污染 | 先在本项目验证一个 Sprint |
| 把项目专有内容塞进模板 | 下个团队无法复用 | L1 留在项目内 |
| 修了模板却没更新 README | 用户不知道新能力 | L3 改动同步文档 |
| 知识库重复堆叠 | 阅读成本上升 | 合并修改优先于新增 |
| 模板演进无版本号 | 难以追溯回归 | 维护 CHANGELOG 与发布标签 |
| 只修项目/模板，不更新知识与来源索引 | 下个团队得到动作却不知道原因，经验无法审计 | 按 §5.1 同时核对来源、L2、L3、验证和发布 |
