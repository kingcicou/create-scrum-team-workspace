# 通用项目开发落地指南

## 1. 总体定位

本模板不是单人全栈脚手架，而是 Scrum 团队协同工作区。它把项目运行、工程设计、代码仓库、质量验证、发布运维、度量改进和知识沉淀放在同一个可追溯系统里。

## 2. 两层结构

| 层级 | 位置 | 作用 |
| --- | --- | --- |
| 项目工作区 | `00_项目导航/` 到 `07_度量改进/` | 文档主事实源、团队运行、决策和证据 |
| 代码仓库 | `10_代码仓库/<RepoName>/` | 应用代码、基础设施脚本、迁移、CI、测试、TeamWork |

## 3. 两个不能混用的分类

### 3.1 项目背景类型

项目背景回答“我们从什么基础开始”，通常在项目启动时确定：

| 类型 | 启动重点 |
| --- | --- |
| 从零项目 | 建立产品、工程和交付基线 |
| 存量重构 | 诊断旧系统，明确保留、替换和迁移边界 |
| 成熟产品迭代 | 延续业务与生产责任，补齐协作和质量门禁 |
| 原型转正 | 保留已验证知识，重建工程质量和运行基线 |

### 3.2 Sprint 仓库策略

仓库策略回答“本 Sprint 如何处理代码”，每次 Planning 都要重新确认：

| 策略 | 使用条件 | 生成器行为 |
| --- | --- | --- |
| `reuse` | 现有仓可持续承载本轮改动 | 只登记现仓，不复制代码、不新建 Git 历史 |
| `import` | 已有代码未规范入库或需要整合 | 建目标仓，先冻结和去敏，再导入 |
| `rewrite` | 新技术栈或架构边界需要独立验证 | 建候选新仓，旧仓并行维护到切换门禁通过 |
| `create` | 完全从零开发 | 建新仓、工程骨架、CI 和角色工作区 |

项目类型不直接决定仓库策略。例如成熟产品通常 `reuse`，但 Sprint 4 经
ADR/Spike 证明需要 Rust 后端和 Svelte 前端时，可以切换为 `rewrite`；项目
背景仍然是“成熟产品迭代”。

## 4. 启动顺序

1. 确认项目背景类型。
2. 在 Sprint Planning 选择仓库策略并更新 `10_代码仓库/00_仓库清单.md`。
3. 校准角色命名和真实成员。
4. 阅读并评审 `05_输入输出管理规范.md`。
5. 校准 `06_团队输入输出总表/00_索引.md`。
6. 完善 Sprint 计划、技术全景和迁移/回退门禁。
7. 确认 Git 模式和 TeamWork 策略。
8. 补齐首批 Backlog、质量策略和发布策略。

## 5. 不要做什么

- 不要在代码仓库再维护一套文档中心。
- 不要把知识库当执行目录。
- 不要把存量项目当从零项目。
- 不要因技术栈变化直接覆盖仍在生产运行的旧仓。
- 不要把同一份代码复制进文档仓，制造第二个代码事实源。
- 不要让 DevOps、QA、UX 工作隐形。
- 不要在没有 DoR 的情况下进入 Sprint Planning。

## 6. 最小验收

- 05/06/SM 三件套已评审。
- 首批 Story 有 AC、依赖和估算。
- 工程设计能支撑并行开发。
- TeamWork / 分支 / Git 身份规则明确。
- 仓库清单能指出现行主仓、候选仓、切换门禁和回退路径。
- DoD 证据链明确。
