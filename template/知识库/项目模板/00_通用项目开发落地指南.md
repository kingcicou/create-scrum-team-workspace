# 通用项目开发落地指南

## 1. 总体定位

本模板不是单人全栈脚手架，而是 Scrum 团队协同工作区。它把项目运行、工程设计、代码仓库、质量验证、发布运维、度量改进和知识沉淀放在同一个可追溯系统里。

## 2. 两层结构

| 层级 | 位置 | 作用 |
| --- | --- | --- |
| 项目工作区 | `00_项目导航/` 到 `07_度量改进/` | 文档主事实源、团队运行、决策和证据 |
| 代码仓库 | `10_代码仓库/<RepoName>/` | 应用代码、基础设施脚本、迁移、CI、测试、TeamWork |

## 3. 项目类型

| 类型 | 使用方式 |
| --- | --- |
| 从零项目 | 直接用模板建立工作区和新仓库 |
| 存量重构 | 先诊断旧系统，模板作为目标结构和协同系统 |
| 成熟产品迭代 | 复用现有仓库，重点补齐 05/06/SM/质量/发布 |
| 原型转正 | 保留业务知识，重建工程质量和协作基线 |

## 4. 启动顺序

1. 确认项目类型。
2. 校准角色命名和真实成员。
3. 阅读并评审 `05_输入输出管理规范.md`。
4. 校准 `06_团队输入输出总表.md`。
5. 按项目类型完善 Sprint 0 计划。
6. 确认代码仓库 Git 模式和 TeamWork 策略。
7. 补齐首批 Backlog、技术全景、质量策略和发布策略。

## 5. 不要做什么

- 不要在代码仓库再维护一套文档中心。
- 不要把知识库当执行目录。
- 不要把存量项目当从零项目。
- 不要让 DevOps、QA、UX 工作隐形。
- 不要在没有 DoR 的情况下进入 Sprint Planning。

## 6. 最小验收

- 05/06/SM 三件套已评审。
- 首批 Story 有 AC、依赖和估算。
- 工程设计能支撑并行开发。
- TeamWork / 分支 / Git 身份规则明确。
- DoD 证据链明确。
