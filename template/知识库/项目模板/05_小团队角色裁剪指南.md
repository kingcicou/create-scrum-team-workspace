# 小团队角色裁剪指南

模板默认 7 个角色槽位（PO / SM / TL / Mid.BE / Sr.FE / Mid.FE / FS）面向 5-7 人团队。
2-4 人可以参考下述帽子合并方式，但当前属于**实验性手工方案**：CLI 仍固定
七个槽位、要求名称/邮箱唯一，并统一创建五个编码角色 worktree，尚未实现
“成员 + 多帽子”的自动生成。

## 1. 原则

- 角色是协作接口（帽子），不是人头。一个人可以戴多顶帽子。
- 保留三条不可省的边界：谁定价值（PO 帽）、谁护流程（SM 帽）、谁兜架构（TL 帽）。
- **帽子稳定，产物按帽子组织**：裁剪只改“帽子→人”的映射，不重写角色卡/签核表/责任表/周期清单（详见 §5）。
- 合并结果可在生成后手工维护；不要直接把同一姓名/邮箱填入多个 CLI 槽位。

## 2. 推荐裁剪档位

| 团队规模 | 建议槽位 | 合并方式 |
| --- | --- | --- |
| 4-5 人 | PO、SM、TL、FS + 1 全栈 | Mid.BE / Mid.FE 合入全栈；SM 可由 TL 兼 |
| 3 人 | PO+SM、TL+BE、FE+FS（各 1 人） | 每人一条价值 / 后端 / 前端主线 |
| 2 人 | 全栈 A（PO+TL+BE）、全栈 B（SM+FE+FS） | 让 PO / SM / TL 三帽分属两人 |

## 3. 兼任红线

- PO 与 SM 尽量不由同一人长期兼任——价值裁决与流程守护会互相牵制。
- 兼任者在文档 `owner` 与 commit `--author` 上仍要如实署真实的人，不因兼任而混淆归属
  （见 `知识库/Scrum/11_角色工作区与Git身份引导规范.md` §8）。
- 隐形工作（CI、环境、设计系统、技术债）无论几人都要进 Backlog 占容量。

## 4. 操作

1. 生成时使用 `--no-worktrees`，避免固定创建五个角色目录。
2. 生成后在角色表和能力矩阵登记“成员 → 帽子”映射，再由 FS 按实际成员手工建 worktree。
3. 在 `07_度量改进/04_能力矩阵.md` 标出每顶帽子的兜底人，避免单点。

> 裁剪的是“人头到帽子的映射”，不是裁掉 Scrum 的帽子。帽子边界在，协作接口就清晰。

## 5. 裁剪后各产物如何适配（多数只改映射，不改结构）

**关键：所有产物都按“帽子”组织，帽子是稳定的；裁剪只改“帽子→人”的映射，不重写产物结构。** 所以改动很小：

| 产物 | 是否需改 | 改什么 | 谁改 |
| --- | --- | --- | --- |
| `roles.config.json` | ⚠️ 手工参考 | 当前 CLI 不读取单角色 worktree 覆盖，也拒绝重复姓名/邮箱 | 初始化者 |
| `02_角色与联系方式` | ✅ | 同一人出现在多行（多帽） | SM |
| `11_角色行动手册` §4 事件台账 | ✅ | 同一人按实际承担的帽子分别追加签核事件；Campaign 可合并通知 | 各成员本人 |
| §1 角色卡 | ⚠️ 不改结构 | 卡按帽子保留不动；兼任者**读自己所有帽子的卡**（并集） | 无需改 |
| §2 责任表 / §3 周期任务清单 | ⚠️ 不改结构 | 处理人列仍写帽子；帽子→人映射由签核表/02 提供 | 无需改 |
| `07_度量改进/04_能力矩阵` | ✅ | 标每顶帽子兜底人，避免单点 | SM |
| 生成器 | ⛔ 尚未自动支持 | 先用 `--no-worktrees`，成员工作区由 FS 手工建立 | FS |

> 自动裁剪的目标模型应是 `members[] + hats[]`。实现和测试完成前，本指南只
> 提供职责设计，不宣称生成器已经自动支持。

## 6. 操作步骤（分场景）

### 场景 A：团队创建之初就是小团队（推荐，最干净）

1. **生成前**先定兼任方案（参 §2 档位），执行生成器时加 `--no-worktrees`。
2. 生成后由 FS 按实际成员建立 worktree，并配置每个工作区的真实 Git 身份。
3. 填 `02_角色与联系方式`（一人多行）；SM 在签核 Campaign 中按实际帽子列出应签角色。
4. 各成员**读自己所有帽子的卡** → 跑通一次真实任务 → **逐帽签核自己那些行**。
5. 在 `04_能力矩阵` 标兜底人。

### 场景 B：中途缩编（有人离开，帽子并给他人）

1. SM 更新角色表和能力矩阵中的帽子映射；FS 调整实际 worktree。
2. 更新 `02` 与签核表“成员”列（接手人补上新帽子行）。
3. 接手人**读新增帽子的卡** → **签核新帽子那几行**（本人提交）。
4. `04_能力矩阵` 更新兜底人；若规范未变，手册基线不必升。

### 场景 C：中途扩编（加人，拆出独立帽子）

1. 在角色表中把兼任帽子分给新人，由 FS 给新人建 `worktree`。
2. 更新 `02`；新人按承担的帽子在 §4 分别追加事件。

> 三场景共同点：**动的都是“帽子→人”映射 + 谁签哪些帽子行**，不改角色卡/责任表/周期清单/生成器。
