# 团队档位选择指南

模板提供 5 档团队预设，帮助团队在项目创建时一次选中最接近现实的启动路线。
运行时使用 `members + scrum + hats + assignments`：成员是人，Scrum 责任与工程帽子是协作接口。
2-7 人均可找到合适档位；一人多帽时签核仍按成员一次完成，不复制身份。

## 1. 原则

- **模板不是让团队在运行中不断裁剪，而是在项目创建时帮助团队一次选中最接近现实的启动路线。**
- 角色是协作接口（帽子），不是人头。一个人可以戴多顶帽子。
- 保留三条不可省的边界：谁定价值（PO 帽）、谁护流程（SM 帽）、谁兜架构（TL 帽）。
- **帽子稳定，任务按成员执行**：任务表同时记录 Owner(memberId) 与责任帽子。
- 不要复制同一人的姓名/邮箱制造多个成员；用 `team.mjs assign` 给同一 memberId 增加帽子。

## 2. 五档团队预设

创建时通过 `--team-profile=<档位>` 选择：

| 档位 | 成员数 | 角色构成 | Worktree 数 | 适用场景 |
| --- | --- | --- | --- | --- |
| `full-7` | 7 | PO / SM / TL / Mid.BE / Sr.FE / Mid.FE / FS | 5 | 完整团队，各司其职 |
| `core` | 3 active | PO / SM / TL（其余 4 人 planned） | 1（TL） | 核心团队先启动，后续扩编 |
| `balanced-5` | 5 | PO / SM / TL / Mid.BE+QA / Mid.FE+FS+DevOps | 3 | 精简团队，保留专业分工 |
| `lean-3` | 3 | PO+SM / TL+BE+QA / FE+FS+DevOps | 2 | 小队三人组，PO 兼 SM |
| `lean-2` | 2 | PO+TL+BE / SM+FE+FS+DevOps+QA | 2 | 极小队双人组 |

> **worktree 规则**：只承担 PO/SM 管理责任的成员不创建 worktree；承担编码帽子的成员创建 worktree。

## 3. 兼任红线

- PO 与 SM 尽量不由同一人长期兼任——价值裁决与流程守护会互相牵制。
  - `lean-3` 的 PO 兼 SM 是显式风险接受，工具会输出 warning。
  - `lean-2` 的 PO≠SM 是强制校验，lead-a 和 lead-b 必须是不同成员。
- 兼任者在文档 `owner` 与 commit `--author` 上仍要如实署真实的人，不因兼任而混淆归属
  （见 `知识库/Scrum/11_角色工作区与Git身份引导规范.md` §8）。
- 隐形工作（CI、环境、设计系统、技术债）无论几人都要进 Backlog 占容量。

## 4. 启动模式与档位搭配

创建时通过 `--startup-mode=<模式>` 选择仓库和 worktree 策略：

| 模式 | 含义 | Git 仓 | Worktree |
| --- | --- | --- | --- |
| `discovery-first` | 先探索，只建文档治理仓 | 文档仓（项目根） | 不创建 |
| `delivery-ready` | 直接交付，双仓模式 | 文档仓 + 独立代码仓 | 按档位创建 |

### 推荐搭配

- **探索型**：`discovery-first + core` — 3 人核心团队先对齐目标和流程
- **直接开发**：`delivery-ready + balanced-5` — 5 人团队直接进入交付
- **小队推荐**：`delivery-ready + lean-2`（或 `lean-3`） — 2-3 人极小队快速交付

> `delivery-ready` 的双仓模式：文档仓 Git 在项目根，代码仓 Git 在 `10_代码仓库/{repo}`，
> 两者历史独立。首签在文档仓运行，不受代码仓影响。

## 5. 向后兼容

| 旧参数 | 新等价 | 说明 |
| --- | --- | --- |
| `--team-stage=core` | `--team-profile=core` | core 档 3 人 active |
| `--team-stage=full` | `--team-profile=full-7` | 全员 7 人 active |

无新参数时等价 `full-7 + discovery-first`（兼容 rc.8 行为）。

## 6. 裁剪后各产物如何适配

**关键：所有产物都按"帽子"组织，帽子是稳定的；档位只改"帽子→人"的映射，不重写产物结构。**

| 产物 | 是否需改 | 改什么 | 谁改 |
| --- | --- | --- | --- |
| `roles.config.json` | ✅ 工具维护 | `members/scrum/hats/assignments` 是团队事实源 | SM |
| `02_角色与联系方式` | ✅ 自动同步 | 一名成员一行，责任列可含多帽 | `team.mjs` |
| 签核 Event | ✅ 自动生成 | 一名成员一次签收 Campaign 中覆盖的全部责任变化 | 成员本人 |
| §1 角色卡 | ⚠️ 不改结构 | 卡按帽子保留不动；兼任者**读自己所有帽子的卡**（并集） | 无需改 |
| §2 责任表 / §3 周期任务清单 | ⚠️ 不改结构 | 处理人列仍写帽子；帽子→人映射由签核表/02 提供 | 无需改 |
| `07_度量改进/04_能力矩阵` | ✅ | 标每顶帽子兜底人，避免单点 | SM |
| `team.mjs` | ✅ | `add/assign/update/set-status/unassign/list/validate` | SM |

## 7. 运行中调整（扩编/缩编）

### 场景 A：团队创建之初就是小团队（推荐，最干净）

1. 用 `--team-profile=lean-3`（或 lean-2）创建工作区，确认初始成员身份。
2. 各成员读自己责任并集，按 Notice **每人签一次**。
3. Sprint 0 仓库决策获批后，按成员建立工作区（delivery-ready 已自动创建）。
4. 在 `04_能力矩阵` 标兜底人。

### 场景 B：中途缩编（有人离开，帽子并给他人）

1. SM 运行 `team.mjs unassign/assign` 调整帽子映射。
2. 工具登记 Change ID 并同步视图；接手人签收新增责任范围。
3. FS/TL 按需要调整实际 worktree 权限，不改历史 Event。
4. `04_能力矩阵` 更新兜底人；若规范未变，手册基线不必升。

### 场景 C：中途扩编（加人，拆出独立帽子）

1. SM 运行 `team.mjs add`，再用 `assign` 分配帽子。
2. 新人补签基础规范与其受影响 Change ID；通过后再取得任务/代码权限。

> 三场景共同点：团队事实只改一处，派生视图由工具同步，历史 Campaign 按创建时身份快照验证。
# 小团队角色裁剪指南

模板初始预设提供 7 个责任槽位，但运行时使用
`members + scrum + hats + assignments`：成员是人，Scrum 责任与工程帽子是协作接口。
2-4 人可让一个成员承担多顶帽子；签核仍按成员一次完成，不复制身份。

## 1. 原则

- 角色是协作接口（帽子），不是人头。一个人可以戴多顶帽子。
- 保留三条不可省的边界：谁定价值（PO 帽）、谁护流程（SM 帽）、谁兜架构（TL 帽）。
- **帽子稳定，任务按成员执行**：任务表同时记录 Owner(memberId) 与责任帽子。
- 不要复制同一人的姓名/邮箱制造多个成员；用 `team.mjs assign` 给同一 memberId 增加帽子。

## 2. 推荐裁剪档位

| 团队规模 | 建议槽位 | 合并方式 |
| --- | --- | --- |
| 4-5 人 | PO、SM、TL、FS + 1 全栈 | Mid.BE / Mid.FE 合入全栈；SM 可由 TL 兼 |
| 3 人 | PO+SM、TL+BE、FE+FS（各 1 人） | 每人一条价值 / 后端 / 前端主线 |
| 2 人 | 全栈 A（PO+TL+BE）、全栈 B（SM+FE+FS） | 让 PO / SM / TL 三帽分属两人 |

## 3. 兼任红线

- PO 与 SM 尽量不由同一人长期兼任——价值裁决与流程守护会互相牵制。
- 兼任者在文档 `owner` 与 commit `--author` 上仍要如实署真实的人，不因兼任而混淆归属
  （见 `知识库/Scrum/11_角色工作区与Git身份引导规范.md` §8）。
- 隐形工作（CI、环境、设计系统、技术债）无论几人都要进 Backlog 占容量。

## 4. 操作

1. 创建时可用 `--team-stage=core`，先由 PO/SM/TL 核心专家启动 Sprint 0。
2. 用 `node tools/team.mjs add ...` 增加真实成员。
3. 用 `node tools/team.mjs assign --member=<id> --hat=<hat>` 分配一顶或多顶帽子。
4. 工具同步联系人/任务视图并登记签核变化；SM 按输出发起增量签核。
5. 代码仓获批后，再按真实成员创建 worktree；一个成员一个工作区，不按帽子复制目录。

> 裁剪的是“人头到帽子的映射”，不是裁掉 Scrum 的帽子。帽子边界在，协作接口就清晰。

## 5. 裁剪后各产物如何适配（多数只改映射，不改结构）

**关键：所有产物都按“帽子”组织，帽子是稳定的；裁剪只改“帽子→人”的映射，不重写产物结构。** 所以改动很小：

| 产物 | 是否需改 | 改什么 | 谁改 |
| --- | --- | --- | --- |
| `roles.config.json` | ✅ 工具维护 | `members/scrum/hats/assignments` 是团队事实源 | SM |
| `02_角色与联系方式` | ✅ 自动同步 | 一名成员一行，责任列可含多帽 | `team.mjs` |
| 签核 Event | ✅ 自动生成 | 一名成员一次签收 Campaign 中覆盖的全部责任变化 | 成员本人 |
| §1 角色卡 | ⚠️ 不改结构 | 卡按帽子保留不动；兼任者**读自己所有帽子的卡**（并集） | 无需改 |
| §2 责任表 / §3 周期任务清单 | ⚠️ 不改结构 | 处理人列仍写帽子；帽子→人映射由签核表/02 提供 | 无需改 |
| `07_度量改进/04_能力矩阵` | ✅ | 标每顶帽子兜底人，避免单点 | SM |
| `team.mjs` | ✅ | `add/assign/update/set-status/unassign/list/validate` | SM |

> PO 与 SM 可以技术上指向同一成员，但工具会告警；这应是小团队的显式风险接受，
> 不是默认推荐。

## 6. 操作步骤（分场景）

### 场景 A：团队创建之初就是小团队（推荐，最干净）

1. 用 core 档创建文档仓，确认初始成员身份。
2. SM 用 `team.mjs` 建成员并分配所有帽子；不得直接复制角色身份。
3. 各成员读自己责任并集，按 Notice **每人签一次**。
4. Sprint 0 仓库决策获批后，按成员建立工作区。
5. 在 `04_能力矩阵` 标兜底人。

### 场景 B：中途缩编（有人离开，帽子并给他人）

1. SM 运行 `team.mjs unassign/assign` 调整帽子映射。
2. 工具登记 Change ID 并同步视图；接手人签收新增责任范围。
3. FS/TL 按需要调整实际 worktree 权限，不改历史 Event。
4. `04_能力矩阵` 更新兜底人；若规范未变，手册基线不必升。

### 场景 C：中途扩编（加人，拆出独立帽子）

1. SM 运行 `team.mjs add`，再用 `assign` 分配帽子。
2. 新人补签基础规范与其受影响 Change ID；通过后再取得任务/代码权限。

> 三场景共同点：团队事实只改一处，派生视图由工具同步，历史 Campaign 按创建时身份快照验证。
