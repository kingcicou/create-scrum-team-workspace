# 文档协作机制迭代计划

**迭代主题：** 解决多角色并发写入共享文档的语义冲突
**决策日期：** 2026-06-29
**适用版本：** 0.4.0

## 1. 起因

实际应用模板后，发现 Markdown-as-database 模式存在严重的并发冲突，已在生产中显化：

| # | 用户反馈 |
| --- | --- |
| 1 | 所有成员更新总表时存在竞争，可能覆盖他人更新 |
| 2 | 两个以上角色协作输出同一份文档时相互影响，最终面目全非 |
| 3 | 交叉评审时全员一拥而上同一文档，无序且无追溯 |

补充识别（用户未直接提到但同样严重）：

- 评审循环无终结
- 回滚责任不明
- 术语命名漂移
- 草稿/正式边界模糊
- 审计追溯困难
- 跨 Sprint 内容稀释
- 评审人轮空
- AI 工具误改
- 文档与代码同分支冲突

详细问题清单见 `Scrum/13_文档协作与并发控制规范.md` §2。

## 2. 评估的候选方案

| 方案 | 评估 | 结论 |
| --- | --- | --- |
| **用户提议**：每角色一个操作区 + SM 统一合并 | 方向对，但 SM 成单点瓶颈，违反"SM 不替决"原则，且形成"草稿区 + 主表"双事实源 | 不采纳 |
| **方案 A**：把 Markdown 迁出，改用 Notion/Linear/Jira | 解决冲突但违反"零依赖"原则，绑定 SaaS 工具 | 不采纳 |
| **方案 B**：纯靠 Git merge 处理冲突 | 现有路径，已经被实践证伪（语义冲突 Git 解决不了） | 不采纳 |
| **方案 C**：多层防御（结构性拆分 + Frontmatter + 评审段 + PR + CODEOWNERS）| 借鉴 ADR/RFC/CODEOWNERS 成熟工程实践，无新工具依赖 | **采纳** |

## 3. 采纳的 9 层防御

| 层 | 机制 | 目标问题 |
| --- | --- | --- |
| 1 | 结构性拆分 | C1, C2, C3 |
| 2 | Frontmatter 唯一所有者 | C1, C2, C5, C6, C8, C11 |
| 3 | 评审专用段 + ACK | C3, C4, C8 |
| 4 | PR + CODEOWNERS | C1, C2, C5, C10, C11 |
| 5 | Draft → Review → Approved 三段流转 | C4, C7, C9 |
| 6 | 06 总表分表落地 | C1（专项） |
| 7 | 三类协作模式（单 owner / 分段 owner / 协作填表） | C2 |
| 8 | 自动化护栏（可选 P2） | C8, C11 |
| 9 | 反模式预防（写进 SM 手册） | 全部 |

完整规范见 `Scrum/13_文档协作与并发控制规范.md`。

## 5. 风险控制

| 风险 | 控制 |
| --- | --- |
| 06 总表拆分破坏既有引用 | 保留 ID（A01..F0x）不变；只是从一个文件移到分文件；外部引用通过索引文件入口跳转 |
| Frontmatter 增加阅读负担 | 仅给"协作类"文档加，Soul 卡/角色卡/学习心得不加 |
| CODEOWNERS 真实账号未填导致 PR 自动指派失败 | 占位符明确标注，模板加注释"Sprint 0 必须替换" |
| SM 手册"反模式"列表过长 | 只加 1 条（"替 owner 做内容合并"），其他详见 13 规范 |
| 三段流转增加流程负担 | Sprint 0 不强制；从 Sprint 1 开始全面执行；Retro 评估调整 |
| 工具未配置 CI 时校验失效 | P2 任务，本轮 0.4.0 不阻塞 |
| Drone/Codeup 等非 GitHub 平台 CODEOWNERS 语法不同 | 模板使用 GitHub 标准语法 + 注释指引；其他平台由团队改写 |
| 团队对 PR + reviewer 流程不熟 | 在 SOP §5 中扩充"文档 PR 流程"小节（已存在框架，本轮不重写） |

## 6. 验证场景

1. **默认生成**：拆分后 06 目录含 7 文件（A/B/C/D/E/F + 索引），每个含 frontmatter
2. **CODEOWNERS 渲染**：含全部 7 个角色占位符，注释明确替换指引
3. **Frontmatter 替换**：`{{OWNER_PO}}` 等占位符被替换为对应角色显示名（如 `Jobs`）
4. **三段流转可用**：草稿区目录在 Story 工作区，评审产物目录在 Sprint 目录，长期目录在主目录
5. **反向回流可执行**：本计划本身即为 02 反向回流指南的"迭代计划"范本
