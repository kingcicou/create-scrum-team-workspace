# 运维与环境

这里沉淀**与具体项目环境/运维工具相关**的稳定经验，是知识库的第三类内容（前两类是 Scrum 方法论和项目模板原则）。

## 1. 定位

| 类型 | 位置 | 例子 |
| --- | --- | --- |
| 决策、可执行脚本 | `06_发布运维/`（项目运行层） | 发布策略、部署架构评估、CI/CD 审计 |
| 当期 Runbook、Story、故障复盘 | `03_迭代运行/<Sprint-x>/` 或 `06_发布运维/` | 一次性事故处理记录 |
| **稳定的环境/工具配置指南** | **本目录** | 系统保活、桌面环境调优、远程开发隧道、镜像源、备份策略 |

> 与 `06_发布运维/` 的边界：**这里是"为什么这样配 + 通用步骤"**，**那里是"项目当期实际部署架构与决策"**。本目录的文档应能被下一个项目直接复用。

## 2. 应包含什么

下列是典型范围（按需新增，不必都建立）：

- **系统层保活**：禁用桌面/系统休眠、防止 SSH 断连、`systemd` 用户服务 linger、`tmux`/`screen` 守护
- **远程开发环境**：VS Code Remote-SSH、Dev Container、隧道/反向代理（如 frp、Cloudflare Tunnel）
- **包管理与镜像**：内网 npm/PyPI 镜像、Docker registry mirror、APT/YUM 源
- **凭据与密钥**：密钥分发约定、`.netrc` / SSH config 模板（**不**写入真实密钥）
- **本地依赖部署**：数据库、Redis、消息队列的本地 Docker compose 范例
- **备份与恢复**：仓库镜像、数据库快照、配置导出脚本
- **观测**：本地日志聚合、journalctl 速查、性能采样命令
- **平台特殊性**：Windows/macOS/Linux 差异、WSL 配置、VM 宿主机层注意事项

## 3. 不应包含什么

| 内容 | 应去哪 |
| --- | --- |
| 当期 Sprint 的发布决策 | `06_发布运维/00_发布策略.md` |
| 一次性故障的临时脚本 | `03_迭代运行/<Sprint-x>/评估产物/<角色>/` |
| 真实密钥、令牌、密码 | **绝不入库**；用密钥管理工具 |
| 客户/业务专有逻辑 | 项目代码仓库或业务文档 |
| 个人工作习惯 | `知识库/Scrum/学习心得_<Name>.md` |

## 4. 命名规范

```text
<主题>_<场景或工具>.md
```

示例：

```text
保活_GNOME桌面禁用休眠.md
保活_systemd用户linger.md
镜像_npm内网镜像配置.md
隧道_VSCode_Remote_SSH反向代理.md
备份_仓库镜像与定时任务.md
```

> 主题词放前面，便于在 IDE 文件树里按场景聚合（所有"保活"在一起）。

## 5. 文档最小骨架

```markdown
# <标题>

**版本：** Vx.y
**适用范围：** <平台/服务/工具版本>
**编写日期：** YYYY-MM-DD

## 背景
（为什么需要本配置，问题症状）

## 修复步骤 / 配置步骤
（编号步骤 + 可复制命令）

## 验证
（验证命令 + 预期输出）

## 对照表 / 速查
（默认值 vs 推荐值）

## 重要提示
（边界条件、踩坑预警、与上层依赖关系）
```

可参考 QFD_Ark 项目的 `运维与环境/Hermes保活配置指南.md`（针对 GNOME VM 上 Gateway 服务保活）作为完整范例。

## 6. 与其它目录的关联

- 文档完成后在 `00_知识库总目录.md` 的"快速入口"表登记。
- 涉及到的可执行脚本进入 `10_代码仓库/00_仓库清单.md` 指定的当前开发仓 `infra/`。
- 故障复盘类的临时记录先放 `03_迭代运行/`，稳定后再回流到这里。

## 7. 维护规则

- 由触发场景的角色起草（如保活由 FS/DevOps、镜像由 TL/FS、桌面调优由实际遇到的成员）。
- 经 Sprint Review/Retro 评审认定"会反复用到"后再沉淀。
- 配置变更时同步更新文档；过时文档应标注或归档，不应静默保留。
