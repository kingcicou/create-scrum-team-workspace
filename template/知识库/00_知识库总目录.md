**原则：** 当期执行产物放项目工作区，稳定经验沉淀到知识库。

## 文档资产地图

### 当前模板规模

v0.9.1 固定生成 `93` 个 Markdown 文件，约 `5,373` 行、`226 KB`。其中：

- `25` 个是知识库参考资料，约 `2,469` 行、`112 KB`，不要求全员阅读或逐项目改写。
- `68` 个是项目入口、目录骨架、Sprint 0、工程/质量/发布槽位和代码说明，
  约 `2,904` 行、`114 KB`。
- 文件被生成只表示路径可用，不表示必须填写、评审或持续更新。

| 区域 | 固定文件数 | 默认 Owner | 使用方式 |
| --- | :---: | --- | --- |
| 项目根 README | 1 | SM / 项目发起人 | 项目总说明 |
| 项目导航 | 18 | SM 统筹；领域内容归对应角色 | 首页、角色行动、协议、总表和 SOP |
| 产品发现 | 6 | PO | 按产品问题选用 |
| 产品待办 | 6 | PO | Backlog/Story/AC 主事实源 |
| Sprint 0 | 9 | SM 管容器；PO/TL/FS/Task Owner 管内容 | 初始模板；后续 Sprint 不必全量复制 |
| 工程设计 | 7 | TL / 资深工程师 | 有真实设计决策时填写 |
| 质量验证 | 5 | QA 帽子角色 / TL | 有测试策略或质量结论时填写 |
| 发布运维 | 4 | FS | 发布、回滚或 Runbook 需要时填写 |
| 度量改进 | 5 | SM | 只保留有决策价值的指标和改进 |
| 代码仓库说明 | 6 | FS / TL | 随仓库策略按需生成或引用 |
| 会议与决策 | 1 | 决策权角色 | 只记录重要结论 |
| 知识库 | 24 | 模板维护者 + 领域 Owner | 按问题查阅，默认只读 |

### 每个项目真正必须维护的最小集合

| 文档/事实 | Owner |
| --- | --- |
| 项目首页 | SM |
| 团队工作协议与角色表 | SM 统筹、全员确认 |
| Product Backlog / Story / AC | PO |
| Sprint 计划 | PO + SM |
| Sprint 任务表：`03_迭代运行/<当前Sprint>/01_Sprint任务表与流程看板.md` §4 | PO/TL/资深成员拆分；Task Owner 更新状态 |
| 仓库清单 | FS |
| 代码、测试和验证证据 | Task Owner + Reviewer |

技术全景、ADR、契约、测试策略和 Runbook 不是每个 Sprint 固定新增；满足文档
创建门时才产生。

### 可预见与不可预见

文档**类别基本可预见，数量和内容不可预见**：

| 动态类别 | 数量规律 | Owner |
| --- | --- | --- |
| 后续 Sprint | 每轮最少计划 + 任务表 2 份 | PO / SM |
| ADR/技术契约 | `0..N`，由真实决策触发 | TL / 对应资深角色 |
| 产品发现/重大变更 | `0..N` | PO |
| Runbook/发布/事故记录 | `0..N`，由重复操作或事故触发 | FS |
| 独立质量报告 | `0..N`，由门禁/审计触发 | QA / TL |
| 稳定经验回流 | 每次 Retro 最多 0-1 项 | 领域 Owner + SM |

估算公式：

```text
项目文档数 ≈ 93 个固定骨架
           + 2 × 后续 Sprint 数
           + 必要 ADR/契约/Runbook/审计文档数
```

无法预见的是业务决策和风险本身，不是文档应该放在哪里。任何新增文档都必须
说明消费者、Owner 和用途，否则更新任务表或现有文档。

## 维护规则

- Sprint Review / Retro 后，由 SM 判断哪些经验值得沉淀。
- 执行期草稿不要直接写入知识库，先放 Sprint 目录。
- 知识库更新需在 `00_项目导航/06_团队输入输出总表/00_索引.md` 中登记。
- 知识库文档要偏原则、机制和复用经验，不要变成当期日报。
