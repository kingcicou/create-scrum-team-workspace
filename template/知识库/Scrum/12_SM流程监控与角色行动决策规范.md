# SM 流程监控与角色行动决策规范

> 阅读分层：本规范供 SM / 流程教练理解判定原理，属于选读材料。团队成员日常只需使用 `00_项目导航/09_SM教练查询与回复模板.md`。

## 1. 目的与边界

本规范帮助 SM / 敏捷教练可靠回答：

1. 项目和 Sprint 当前处于什么阶段？
2. 哪条工作流正在推进、等待或阻塞？
3. 某角色现在有哪些必做、等待、可先行和协作活动？
4. 什么情况下教练必须停止推断并要求更新事实源？

SM 管控的是流程透明度、反馈和清障机制，不是替成员派活，也不替 PO、TL、FS 做专业决策。

## 2. 三层控制体系

| 层级 | 主事实源 | 回答的问题 |
| --- | --- | --- |
| 项目级产物账本 | `00_项目导航/06_团队输入输出总表/00_索引.md` | 产物在哪里、由谁负责、依赖和评审状态是什么 |
| Sprint 任务与流程控制 | `03_迭代运行/<Sprint>/01_Sprint任务表与流程看板.md` | 任务、复杂度、状态、依赖、证据和门禁是什么 |
| Sprint 关闭控制 | `03_迭代运行/<Sprint>/07_Sprint关闭与准入检查表.md` | 时间盒、目标、治理闭环和下一轮准入结论是什么 |
| 教练判定机制 | 本规范与 SM 作战手册 | 如何从事实源得出行动建议、何时干预 |

看板/Backlog 是 Story 优先级事实源，Git/PR/CI 是代码事实源，风险台账是障碍事实源。流程监控台只保存摘要和链接，不复制完整内容。

## 3. 阶段表达

Scrum 团队不是瀑布式地全员同步跨阶段。教练应使用：

```text
当前主阶段 + 各工作流阶段
```

推荐阶段词典：

| 阶段 | 关键问题 | 通过条件 |
| --- | --- | --- |
| 产品发现 | 为什么做、为谁做 | 愿景、场景和价值假设可供 Backlog 消费 |
| Backlog 准备 | 做什么、如何验收 | Story 满足 DoR，AC 和优先级明确 |
| 工程设计 | 如何安全实现 | 关键 ADR、API、数据和体验契约足以开工 |
| 编码实现 | 如何形成可运行增量 | 分支、代码、自测和 Review 证据持续形成 |
| 集成验证 | 各部分能否共同工作 | PR、CI、集成测试和缺陷处置满足门禁 |
| 验收发布 | 是否交付价值且可回退 | PO 验收、发布策略、回滚和 Runbook 就绪 |
| 回顾改进 | 下一轮如何更好 | 1-3 项改进进入台账并有责任人 |

每条工作流单独标记：`未开放`、`待开始`、`进行中`、`等待输入`、`阻塞`、`门禁通过`。

## 4. 依赖、时序与并行判定

教练必须先把候选活动组织成依赖图，再回答角色行动。关系分为：

| 关系 | 含义 | 时间线处理 |
| --- | --- | --- |
| 硬依赖 | 前项未完成，后项不能安全开始 | 后项放入下一时序层 |
| 软依赖 | 草稿可开始，定版必须等待 | 当前层标记“可先行”，汇合前复核 |
| 独立并行 | 无进行中的前置依赖 | 放入同一时序层的不同泳道 |
| 反馈依赖 | Review 可能让前项返工 | 保留回边和复核点 |
| 决策门禁 | 需要 PO/TL/FS 授权或取舍 | 建立汇合门，列出决策人 |

建议表达：

```text
L0 现在并行 -> L1 随后 -> G1 汇合门 -> L2 下一阶段
```

关键路径是“影响 Sprint Goal 最早达成的因果链”，不是最长的个人任务清单。
不在关键路径上的独立活动仍可并行，但不应挤占清障和完成在制品的容量。

## 5. 任务动作分类

| 分类 | 判定条件 | 教练回应 |
| --- | --- | --- |
| 必做 | 属于 Sprint Goal；输入齐备；优先级高；角色有容量 | 明确任务、完成证据和截止点 |
| 等待输入 | 硬依赖未完成；继续会造成返工或越权决策 | 指明缺失输入、负责人和预计解除时间 |
| 可提前先行 | 低耦合、可逆、不会固化未决方案 | 推荐 Mock、用例、Spike、清单、骨架或预研 |
| 协作清障 | 能通过 Review、配对、澄清或补证据解除团队瓶颈 | 指向被帮助对象和清障结果 |
| 暂停/升级 | 触碰价值、架构、安全、质量或发布门禁 | 停止不可逆实施，交还对应决策人 |

“等待输入”不等于“无事可做”。教练应继续检查可先行和协作清障活动，但不能为了保持忙碌而制造无价值 WIP。外部环境缺失但仍可编码或形成操作说明时，标记“待环境验证”，不升级为阻塞。

## 6. 教练判定算法

```text
1. 检查快照新鲜度
2. 确认 Sprint Goal、Day 和范围
3. 按工作流检查进入条件、依赖和门禁
4. 从 Backlog/总表/风险/PR 中收集候选活动及前置依赖
5. 标记硬依赖、软依赖、独立并行、反馈和决策门禁
6. 按拓扑层次形成“现在并行 -> 随后 -> 汇合门”时间线
7. 识别影响 Sprint Goal 的关键路径
8. 检查任务 Owner 当前 WIP 与能力边界
9. 按必做、等待、可先行、协作、暂停分类
10. 给出完成证据、解锁对象、责任人和预计时间
```

排序原则：

1. 保护 Sprint Goal 和生产稳定性。
2. 先解除团队瓶颈，再拉取新工作。
3. 先完成在制品，再增加 WIP。
4. 优先短反馈、可验证和可逆行动。
5. 不以“每个人都很忙”作为健康标准。

## 7. 信息新鲜度与停止推断

| 信号 | 阈值 | 行动 |
| --- | --- | --- |
| 总表或看板未更新 | 超过 1 个工作日 | 标记快照不可靠，要求责任人更新 |
| 阻塞未登记 | 超过半天 | 进入风险与障碍台账 |
| CI 红灯 | 超过 2 小时 | 升级为 Sprint 风险 |
| 单人 WIP | 超过 2 项 | 停止拉新，推动完成或协作 |
| 待验收 | 超过 2 天 | 提醒 PO 验收或重新拆分 |
| 两个事实源冲突 | 发现即处理 | 暂停结论，确认主事实源 |

满足以下任一条件时，教练必须回答“当前状态无法可靠判断”：

- 关键事实源超过约定时间未更新。
- Backlog 优先级与 Sprint 计划冲突。
- 总表状态与真实产物、PR 或 CI 不一致。
- 依赖责任人和预计解除时间均未知。
- 请求涉及 PO/TL/FS 尚未作出的专业决策。

## 8. 对外输出

成员状态包、SM 确认与纠偏，以及以下三类输出均以 `00_项目导航/09_SM教练查询与回复模板.md` 为准：

| 场景 | 输出 |
| --- | --- |
| 成员主动报告等待、阻塞或交接 | 成员状态包 + SM 确认 |
| 站会后、关键变化、群内协调 | 群聊快报 |
| Sprint 中期、Review 前、复盘 | 流程全景 |
| 查询单个角色或任务 | 单角色状态卡 |
| 事实不足或来源过期 | 无法判断模板，点名缺失输入 |
| 发起、纠偏或关闭角色签核 | 签核通知 |

SM 先识别问题类型，再选择模板；不得对所有问题复制同一份全景长回复。
本规范只解释 SM 如何判定，不再重复操作模板。输出必须区分事实与推测；
没有依据的内容标为“待确认”。

### 8.1 流程通知与签核编排

SM 不只回答状态，也负责把已裁决的流程动作翻译成简洁、可执行的通知。通知至少
说明目的、范围、对象、动作、截止时间、完成证据和答疑入口。

角色手册签核中，SM 对“登记 Change ID、建立 Campaign、确定范围、发通知、
核验事件、宣布闭环”负唯一编排责任；成员只阅读受影响内容并追加本人 Event。
SM 不得把编排任务转给开发成员。

签核必须区分：

- **当前有效性**：是否已覆盖当前受影响 Change ID。
- **历史完整性**：旧版本是否有当时的独立事件与 Git 证据。

跨版本 catch-up 可以一次覆盖多个 Change ID，但不能倒推为旧版本发布时已签。
全量重基线可恢复当前有效性，不得抹去重基线前的历史缺口。Event ID 的提交
证据由 Git 反查，避免在同一提交中手填自身 hash。

`auto` 证据用 `git blame` 取签核行的当前作者（而非 `git log -S` 的首现提交），
与“成员”列比对：作者≠成员则标「⚠️ 疑似代签」。这是「Git 署名一致性检测」，
用于发现代提与预铺，不等同于不可伪造的身份认证。只有非异常证据才计入已验证覆盖；
异常（疑似代签/无效）事件覆盖的 Change ID 必须继续显示待重签，不得被后续本人事件掩盖。

新 `auto` 事件在 Git 提交前属于“待提交/验证”，不得计入当前有效覆盖；只有
`LEGACY + unverified` 可按迁移兼容规则计入，并继续标记历史缺口。Campaign 只有
在审计待处理为 0 时才能关闭。关闭后发现异常，不倒改旧记录，由 SM 建立
`corrective` Campaign，仅通知受影响角色完成纠偏。

`event-files-v1` 起，`.team/signoffs/` 是唯一事实源：

- SM 用 `prepare` 创建并提交 Campaign 文件，assignment 精确到角色。
- 成员用 `sign` 生成本人 Event 文件；姓名、邮箱、日期和覆盖由工具写入。
- Event 首次提交后不可修改；首次作者与邮箱必须匹配 `roles.config.json`。
- `status` 未归零时 `close` 必须失败；Closure 文件只能由 SM 首次提交。
- 手册中的旧事件/批次表只读兼容，不再人工追加。

当签核机制本身发生实质变化时，必须登记影响全员的新 Change ID，并让新
Campaign 同时覆盖旧缺口和新规则，不能只补签旧基线。

## 9. 维护责任

| 内容 | 更新责任 |
| --- | --- |
| Story 优先级、Sprint 范围 | PO |
| 架构、契约、技术门禁 | TL |
| PR、CI、集成和发布状态 | FS / 提交人 |
| 测试证据、质量门禁 | QA 帽子角色 / TL |
| 风险、预警、监控台快照 | SM |
| 任务状态和完成证据 | 对应 Task Owner |

SM 可以提醒和校验，但不应代替责任人填写专业结论。

Sprint结束时，SM按`14_Sprint关闭与证据治理规范.md`组织证据复算和遗留处置，但不把门禁blocked改写成“基本通过”。

## 10. 反模式

| 反模式 | 后果 | 修正 |
| --- | --- | --- |
| 再建角色日报或第二份任务表 | 两套状态迅速不一致 | 只更新 Sprint 任务执行表 |
| 把阶段当作全员同步瀑布 | 并行工作被错误暂停 | 使用主阶段 + 工作流阶段 |
| 为了利用率给等待成员随意派活 | WIP 和返工增加 | 优先可逆准备和协作清障 |
| 教练替 PO/TL 做决定 | 角色责任失真 | 明确暂停/升级和决策人 |
| 信息过期仍给确定答案 | 团队依据错误信息行动 | 停止推断并要求更新 |
| 按角色逐个罗列但没有因果链 | 看不出先后、并行与关键路径 | 先输出时间线和汇合门，再输出角色行动 |
| 把平台权限/环境安装一律标红 | 团队可并行工作也被暂停 | 区分待环境验证与真实目标阻塞 |
| 把签核编排转给某个开发成员 | 范围误派、漏签且无人验收 | SM 建 Change/Campaign、逐人通知、核验并闭环 |
| 只保存每人最新基线 | 无法区分旧版当时签核与后续补签 | 追加 Event，显式记录覆盖 Change ID |
| 在签核行手填自身 commit | 产生自引用和错误仓库哈希 | 用 Event ID 由审计器从 Git 反查 |
| 全量重签后宣称旧历史完整 | 用当前确认伪造过去证据 | 当前有效性与历史完整性分别报告 |
| 已关闭批次仍有待处理却不发新任务 | 审计能发现问题但无人负责收口 | 保留旧批次并建立 corrective Campaign |
| 他人预铺事件后本人改空格取得 blame | 当前行署名掩盖首次创建者 | 独立不可变 Event 文件并校验首次提交 |
| 规范升级后只补旧缺口 | 成员签了旧规则却不会使用新机制 | 新 Change ID 影响全员，Campaign 覆盖新基线 |
