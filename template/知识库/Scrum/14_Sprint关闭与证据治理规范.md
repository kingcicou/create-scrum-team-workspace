# Sprint 关闭与证据治理规范

> 本文提炼自 QFD_Ark `14_Sprint0外部评审教训总结.md`，保留其中六条可跨项目复用的教训。

## 1. Sprint 结束不等于全部 Done

Sprint 到达结束日期就结束，未完成事项回到 Backlog、明确结转或取消，不能通过延长 Sprint 隐藏遗留。

关闭时分别回答：

| 维度 | 可选结论 |
| --- | --- |
| 时间盒 | open / closed |
| Sprint Goal | achieved / partial / missed |
| 遗留处置 | closed / carry-over / cancelled |
| 下一 Sprint 准入 | blocked / passed / waived |

推荐表达：

```text
Sprint 0：时间盒已关闭 / Goal 已达成 /
遗留已结转 / Sprint 1 准入 blocked
```

## 2. Approve、门禁通过和交付不能混用

| 层级 | 含义 | 典型证据 |
| --- | --- | --- |
| 方案 approved | 评审人确认方案合理 | 评审 ACK |
| 门禁 passed | 达到进入下一阶段的客观标准 | 测试、CI、检查结果 |
| 已交付 | 消费者能够实际使用 | 部署、验收或发布结果 |

方案写完或文档 approved，不代表 CI 已运行，也不代表功能已交付。

## 3. 正式文档状态必须一致

正式协作文档使用：

```yaml
owner: <单值>
reviewers: [按风险指定]
status: draft|review|approved|locked
version: <MAJOR.MINOR>
last-updated: <日期>
```

总表、Frontmatter、正文和评审记录必须表达同一状态。

README、目录、持续维护台账可以使用替代证据，不机械强制 Frontmatter，但应写明豁免理由。

## 4. 事实冲突先裁决再同步

发现总表、文档、PR、CI或实际制品冲突时：

1. **冻结**：暂停冲突项状态变化和门禁结论。
2. **取证**：检查评审记录、自动化结果、实际制品和验收结果。
3. **裁决**：由对应专业责任人给出结论，SM不替代专业判断。
4. **同步**：修正所有受影响事实源。

检查应由事件触发：状态变更、门禁判定、Review提交前。Daily不演化成文档审计会。

## 5. 门禁标准必须提前写硬

门禁在 Planning 时定义，至少包含：

- 客观通过条件。
- 可接受证据。
- 决策人。
- 不通过时的处理。

waiver 是有期限的风险接受，不等于 passed；至少写明原因、风险、补偿措施、owner、批准人和失效时间。

## 6. 统计必须可复算

不要只报一个“完成率”。至少区分：

- 核心范围与追加范围。
- 已关闭、持续维护和未完成。
- Sprint Goal 结论与门禁结论。

同一事项只能进入一个统计单元，并保留ID或路径明细。没有证据时标“待确认”，不能为了数字好看推断完成。

## 7. SM 的责任边界

SM不替owner写专业结论，但必须校验：

- 总表与实际产物一致。
- 总结数字可以从明细复算。
- 门禁有客观证据。
- 未完成事项有明确去向。

事实不足时，SM应回答“当前无法可靠判断”，而不是补一个乐观结论。

## 8. Sprint 关闭最小流程

1. 完成 Sprint Review 和 Retro。
2. 核对 Sprint Goal，并列出证据。
3. 对未完成项选择关闭、结转或取消。
4. 分别判定质量、CI、发布等门禁。
5. 检查任务证据和“例外与裁决”；`open` 例外不得关闭，carry-over 必须有 Owner 和去向。
6. 锁定 Sprint 文档，更新项目首页、迭代日历和角色工作入口。
7. 由 PO/TL/FS 按职责确认结论。
8. 需要时建立 annotated tag，消息写明 Goal、门禁和遗留。

tag 是关闭时点的基线，不等于 release，也不表示全部事项 Done。已推送 tag 不应移动。

## 9. 计划时间与实际时间

- 计划周期在 Planning 时锁定，不因提前完成而回写。
- 工作完成时间表示 Goal 所需工作达到约定结果。
- 正式关闭时间表示 Review、Retro、遗留处置和事实同步完成。
- 经常提前完成时调整下一 Sprint 容量或范围，不通过改写历史计划制造准确感。

## 10. 平台无关的 CI 最小证据

流水线平台和字段可以不同，但结论至少包含：

```text
平台/目标环境｜运行 ID 或 URL｜commit SHA/分支｜触发时间
结果｜验证人｜关键 Gate 摘要
```

平台没有统一运行 ID 或 URL 时，可用 Job ID、日志文件、截图或日志摘要哈希
替代。证据格式不同不等于未完成，但必须能够关联到提交、时间、结果和验证人。

CI 不是每个 Sprint 都要重建。仓库、技术栈、质量门禁或平台发生变化时才建立
配置/迁移 Task 或 Spike；稳定流水线只需要按风险做触发验证。

## 11. 单一事实源归档与关闭后修正

- Sprint 正文留在运行目录并锁定，归档目录只登记关闭时间、tag 和权威链接；
  或整体移动 Sprint 目录。两种方式二选一，禁止复制部分文件形成双事实源。
- 已推送 tag 后发现事实错误，用普通 post-close 修正提交处理，不移动 tag。
- 修正只处理日期、状态、证据链接和当前入口等事实，不借机重写历史决策。

## 12. 最小自检

- [ ] 正式文档状态与总表一致。
- [ ] 测试或CI结论来自实际运行，不是未来计划。
- [ ] 门禁定义没有在结果出来后放宽。
- [ ] 总结数字可从ID或路径明细复算。
- [ ] 未完成事项已进入Backlog、台账或取消记录。
- [ ] Sprint关闭与下一Sprint准入分别表述。
- [ ] 首页、日历和角色入口已切换，归档没有第二套正文。
- [ ] “已完成”任务有证据；未运行的平台验证没有被改写为完成。
- [ ] 启动、分支、身份、评审、目标或关闭偏离均已裁决。
