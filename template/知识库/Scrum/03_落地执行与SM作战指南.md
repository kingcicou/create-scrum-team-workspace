# 落地执行与 SM 作战指南

## 1. 执行系统的核心

Scrum 落地不是多开几场会，而是建立一套“输入 -> 输出 -> 证据 -> 反馈 -> 改进”的闭环。

本模板中的执行核心：

| 机制 | 文件 |
| --- | --- |
| 团队规则 | `00_项目导航/01_团队工作协议.md` |
| 输入输出边界 | `00_项目导航/05_输入输出管理规范.md` |
| 团队协作雷达 | `00_项目导航/06_团队输入输出总表.md` |
| SM 每日作战 | `00_项目导航/SM_作战手册_Sutherland.md` |
| Sprint 运行 | `03_迭代运行/` |
| 改进闭环 | `07_度量改进/03_改进项台账.md` |

## 2. 输入输出管理原则

- 规范先于产出。
- 谁产出、谁更新。
- 总表同步是产出动作的一部分。
- Draft/WIP 不进长期目录。
- 可执行制品进代码仓库，说明和决策进工作区。
- 代码仓库不维护第二套文档中心。

## 3. SM 每日检查

```text
站会前：
□ 看板状态
□ 总表 P0 产物状态
□ 阻塞是否登记
□ CI/验证是否红灯
□ Story 是否积压未验收

站会后：
□ 更新障碍台账
□ 提醒未更新总表的产出人
□ 协调清障
□ 自检：没有派活、考核、替别人决策
```

## 4. 触发式干预

| 信号 | SM 动作 |
| --- | --- |
| 阻塞超过半天 | 登记障碍并清障 |
| CI 红灯超过 2 小时 | 标记风险，协调 FS/TL/提交人 |
| 总表和真实产物不一致 | 要求产出人更新 |
| 插单进入 Sprint | 推动 PO 做替换取舍 |
| 同一文档两套事实源 | 确认主事实源，另一处改链接 |
| TeamWork 被误提交 | 移除提交并检查 `.gitignore` |
| Retro 改进项过多 | 收敛到 1-3 个 |

## 5. 总表用法

总表不是为了“汇报给管理者”，而是为了回答：

- 谁正在产出什么？
- 产物在哪里？
- 谁要消费它？
- 有什么依赖？
- 是否评审过？
- 是否阻塞？

## 6. 评审机制

| 文档类型 | 评审人 |
| --- | --- |
| 产品/Backlog | PO + TL/QA |
| 架构/API/数据 | TL + 受影响成员 |
| 质量/测试 | QA + TL |
| 发布/运维 | FS + TL |
| 管理流程 | SM + PO |

## 7. 反模式

- 总表事后补账。
- Daily 变成逐人汇报。
- SM 变成项目经理。
- 需求不满足 DoR 仍被塞进 Sprint。
- 代码写完就 Done。
- Review 展示 PPT 而不展示真实增量。
- Retro 没有负责人和截止时间。
