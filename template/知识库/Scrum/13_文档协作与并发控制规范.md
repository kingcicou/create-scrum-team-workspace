# 文档协作与并发控制规范

## 1. 目标与范围

本规范解决团队在 Markdown-as-database 模式下的**共享可变状态并发写入冲突**。它不是工具，是协议——基于 Git 平台、Frontmatter 与 CODEOWNERS 形成多层防御。

适用对象：所有进入团队协作目录的 `.md` 文档，特别是：

- `00_项目导航/` 中由 SM 协调的项目级账本
- `01_产品发现/`、`02_产品待办/` 中由 PO 主导的产品文档
- `04_工程设计/` 中由 TL 主导的架构/契约/数据模型文档
- `05_质量验证/`、`06_发布运维/` 中由 QA / FS 主导的质量与发布文档
- `知识库/` 中的稳定经验

个人草稿、Soul 卡、心得不强制使用本规范。

## 2. 三类共享可变状态冲突（问题列表）

| # | 冲突类型 | 典型表现 |
| --- | --- | --- |
| C1 | 总表竞争写入 | 多人同时改 `06_团队输入输出总表`，后保存者覆盖前者 |
| C2 | 协作设计文档稀释 | 多人在同一份评估/ADR 上互相覆盖，最终文档风格、术语、结构面目全非 |
| C3 | 评审拥挤 | 多人同时在一份待评审文档上批注、修改、加意见，无追溯 |
| C4 | 评审循环无终结 | 文档标"待评审"无限期，没人知道何时定版 |
| C5 | 回滚责任不明 | 改坏了找谁恢复，没人愿意做"破坏者" |
| C6 | 术语命名漂移 | 同一概念多种说法，后来人困惑 |
| C7 | 草稿/正式边界模糊 | 在长期目录改还是 Sprint 草稿区改，半途接手谁负责 |
| C8 | 审计追溯困难 | "谁在何时改了什么、为什么"无元数据 |
| C9 | 跨 Sprint 内容稀释 | 上 Sprint 结论被下 Sprint 的人无意覆盖 |
| C10 | 评审人轮空 | 标"待 PO 评审"但 PO 不知道，无通知/认领机制 |
| C11 | AI 助手误改 | 自动化工具按上下文重写，覆盖另一角色的细微调整 |
| C12 | 文档与代码同分支冲突 | 编码角色在 feature 分支改文档，文档主线又在 main 推进 |

## 3. 多层防御总览

```text
预防（结构性） → 隔离（Frontmatter） → 协议（评审段 / 三段流转）
                                       ↓
                              平台护栏（PR + CODEOWNERS）
                                       ↓
                                  自动化（CI / hook）
```

| 层 | 机制 | 解决问题 |
| --- | --- | --- |
| 1 | 结构性拆分 | C1, C2, C3 |
| 2 | Frontmatter 唯一所有者 | C1, C2, C5, C6, C8, C11 |
| 3 | 评审专用段 + ACK | C3, C4, C8 |
| 4 | PR + CODEOWNERS | C1, C2, C5, C10, C11 |
| 5 | Draft → Review → Approved 三段流转 | C4, C7, C9 |
| 6 | 06 总表分表落地 | C1（专项） |
| 7 | 三类协作模式（单 owner / 分段 owner / 协作填表） | C2 |
| 8 | 自动化护栏（CI / pre-commit，可选） | C8, C11 |
| 9 | 反模式预防（写进 SM 手册） | 全部 |

## 4. 层 1 — 结构性拆分

把"一份大文档"拆为多份小文件，让冲突从语义层下降到文件层。

| 原 | 改 |
| --- | --- |
| `06_团队输入输出总表/00_索引.md` 单文件 | `06_团队输入输出总表/` 目录 + 按角色 6 分表 |
| 一份 ADR | 一个决策一个文件（已经是约定） |
| 综合评估报告 | 按章节或角色拆 |
| Story 评审 | 一个 Story 一个工作区 |

## 5. 层 2 — Frontmatter 唯一所有者

所有进入本规范范围的文档头部必须有 YAML frontmatter：

```yaml
---
owner: PO              # 唯一编辑权（写正文）
reviewers: [TL, SM]    # 评审人（写评审段）
status: draft          # draft | review | approved | locked
version: 0.1
last-updated: 2026-06-29
---
```

字段语义：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `owner` | ✅ | 唯一字符串，对应 02_角色与联系方式 中的角色名。只有 owner 能改正文 |
| `reviewers` | ✅ | 数组，建议 1-3 人。其他人可看可评，但不进 reviewers 名单 |
| `status` | ✅ | 流转状态，见 §8 |
| `version` | ✅ | SemVer 简化版 `MAJOR.MINOR`；status 跨段时 bump |
| `last-updated` | ✅ | 最近一次正文更新日期 |

**铁律**：

- 只有 `owner` 能改正文
- 其他人只能在文档末尾"评审意见追加"段加新条目（见 §6）
- `status: locked` 表示冻结，必须 bump version 才能改

## 6. 层 3 — 评审专用段 + ACK 闭环

文档末尾固定保留：

```markdown
## 评审意见追加

### Ritchie · 2026-06-29 14:30
- §3 第 2 段建议改 X 为 Y，理由：……
- §4 表格少了 Z 列

### Norman · 2026-06-29 15:00
- 同意 Ritchie 第 1 条
- §5 我补充：……
```

owner 消化后**只在正文修改**，并在 ACK 区追加：

```markdown
## 评审 ACK

- [x] Ritchie 2026-06-29 14:30 全采纳到 §3.2 / §4
- [x] Norman 2026-06-29 15:00 部分采纳，理由：……
- [ ] 待 Fowler 评审
```

**协议**：

- 评审意见追加只增不改；新条目必须有作者 + 时间
- ACK 必须配对评审意见，未消化的不能 approve
- 评审窗口默认 2 个工作日；超时 SM 升级到 Sprint 风险
- `status: review` 期间，评审段开放；`status: approved` 后归档评审段到 `评审历史/<date>.md`

## 7. 层 4 — PR + CODEOWNERS

- 所有进入本规范范围的文档改动**必须**走 PR
- 仓库根 `.github/CODEOWNERS` 文件按路径自动指派 reviewer
- PR 必须 frontmatter 的 `reviewers` 中至少 1 人 approve 才能合并
- 文档 PR 走 `main`，代码 PR 走 `sprint-<n>`（与 SOP 一致）

`CODEOWNERS` 示例：

```text
# 全局兜底
*  @<sm-github>

# 06 总表分段
00_项目导航/06_团队输入输出总表/A_*  @<sm-github>
00_项目导航/06_团队输入输出总表/B_*  @<po-github>
00_项目导航/06_团队输入输出总表/C_*  @<tl-github>

# 工程设计
04_工程设计/01_ADR/  @<tl-github>
04_工程设计/02_API契约/  @<tl-github> @<midbe-github>

# 产品发现
01_产品发现/  @<po-github>
02_产品待办/  @<po-github>
```

> 团队在 Sprint 0 把 `<*-github>` 替换为真实 GitHub/GitLab/Codeup 用户名。

## 8. 层 5 — Draft → Review → Approved 三段流转

```text
Story 工作区/<角色>/草稿_xxx.md           ← owner 私有，无 PR
       ↓ 内容稳定后提 PR (status: review)
Sprint 目录/评估产物/<主题>_<owner>.md     ← reviewers approve
       ↓ status: approved
长期目录/<主题>.md                          ← status: approved
       ↓ Sprint 结束归档评审段
长期目录/<主题>.md (status: approved, 评审段归档)
```

**铁律**：

- `draft` 状态在草稿区，**不要**直接写长期目录
- `review` 状态进入评审段，2 个工作日窗口
- `approved` 后修改要 bump version 且重新走 review
- `locked` 用于发布版（如 V1.0）冻结快照，必须开新 version 才能改

## 9. 层 6 — 06 总表分表落地

| 原文件 | 拆为 |
| --- | --- |
| `06_团队输入输出总表/00_索引.md` | `06_团队输入输出总表/` 目录 |
| 全员都写同一文件 | 每个分表只由对应主责写入，零冲突 |

```text
06_团队输入输出总表/
├── 00_索引.md             owner: SM       聚合视图、规则与状态总览
├── A_项目管理.md          owner: SM       项目级流程产物（A01-A0x）
├── B_产品发现.md          owner: PO       产品愿景、Backlog（B01-B0x）
├── C_工程设计.md          owner: TL       架构、ADR、API、数据（C01-C0x）
├── D_质量验证.md          owner: QA-Mid.BE 与 Mid.FE 共担（D01-D0x）
├── E_发布运维.md          owner: FS       代码仓、CI、发布（E01-E0x）
└── F_度量改进.md          owner: SM       度量与改进项（F01-F0x），允许评审段追加
```

**特殊规则**：

- A/B/C/E 严格单 owner，其他人**不直接改**
- D 由 Mid.BE 与 Mid.FE 联合管理（API QA + E2E QA），通过 frontmatter `owner: [Mid.BE, Mid.FE]` 写双 owner，但每行有 row-owner
- F 度量改进**允许全员通过评审段提行**，SM 汇入主表
- 索引 `00_索引.md` 只读，由 SM 在站会前手工或脚本聚合（不强制自动化）

## 10. 层 7 — 三类协作模式

| 场景 | 模式 | 例子 |
| --- | --- | --- |
| **单 owner 文档** | 一人写、多人评 | ADR、Story 工作区、单角色产出（如 PO 写愿景、TL 写技术全景） |
| **分段 owner 文档** | 多人按章节分段，每段标 owner | 综合评估报告、跨域设计 |
| **协作填表** | 拆为多个文件 | 06 总表、能力矩阵（每人一行）、风险台账 |

**分段 owner 写法**：

```markdown
## §3 后端架构 (owner: TL · last-updated: 2026-06-29)

正文……

## §4 前端架构 (owner: Sr.FE · last-updated: 2026-06-29)

正文……
```

每段只有该 owner 能改，其他人到评审段反馈。

## 11. 层 8 — 自动化护栏（可选 P2）

- pre-commit hook：frontmatter 完整性校验
- CI：评审段所有条目都有 ACK 才能 merge
- CI：markdown 链接有效性
- pre-commit hook：提醒"你改的不是自己 owned 的文件"

这些是锦上添花，不阻塞落地。

## 12. 层 9 — 反模式预防

| 反模式 | 后果 | 修正 |
| --- | --- | --- |
| SM 替团队做内容合并 | SM 成单点瓶颈，角色失责 | SM 只做流程校验，内容由 owner 守 |
| 长期目录直接改 | 草稿混入主事实源 | 走三段流转 |
| 隔空覆盖 | 工作丢失 | PR + reviewer 必须 |
| 没有 frontmatter | 责任不明 | CI 校验阻塞 |
| 评审无限循环 | 决策瘫痪 | 2 天评审窗口 + 强制收口 |
| AI 工具自由改 | 隐式覆盖 | 工具改动也要 PR + owner ACK |
| 跨 Sprint 覆盖历史 | 决策遗失 | `version` bump + `last-updated` 记录 |
| 在评审段直接改正文 | ACK 链断裂 | 评审只追加，正文只 owner 改 |
| 一份大文档全员同写 | C1/C2 冲突 | 按 §4/§10 拆分 |
| reviewers 列表过长 | 评审拥堵 | 限 ≤ 3 人，其他人留评论但不阻塞 merge |
