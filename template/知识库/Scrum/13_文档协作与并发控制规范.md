# 文档协作与并发控制规范

> 本规范是冲突升级工具，不是所有文档的默认流程。普通任务优先更新 Sprint
> 任务表；只有关键或高冲突产物才逐级增加文档治理。

## 1. 先判断是否需要文档

满足以下任一条件才新建正式文档：

1. 有需要长期追溯的决策、取舍、风险或后果。
2. 是两个以上角色共同消费的契约。
3. 是会重复执行且出错成本高的 Runbook。
4. 法务、安全、验收或审计要求独立证据。

普通实现、测试过程、个人进度和一次性操作默认只保留代码、测试结果和任务
状态。完整门槛见 `00_项目导航/05_输入输出管理规范.md`。

## 2. 三级治理

| 级别 | 适用场景 | 必需控制 | 不要求 |
| --- | --- | --- | --- |
| L0 默认 | 任务状态、个人说明、低冲突记录 | 明确 Owner；任务表留证据 | Frontmatter、PR、CODEOWNERS |
| L1 关键产物 | ADR、契约、发布/回滚、跨角色设计 | 单一 Owner；1-2 Reviewer；明确结论 | 自动化审批、复杂状态流 |
| L2 高冲突 | 多人频繁修改同一正式产物 | 分区所有权；版本/状态；合并协议 | 指定某一种 Git 平台 |

从 L0 升级的触发条件：

- 两人同时改正文并发生一次真实冲突。
- 文档结论直接影响多个模块或发布门禁。
- 评审意见无法确认由谁处理。
- 监管、审计或外部协作要求可追溯审批。

问题消失后可以降级，不把临时治理永久化。

## 3. 角色边界

| 角色 | 责任 |
| --- | --- |
| Owner | 保持正文一致，采纳或拒绝意见，发布最终结论 |
| Reviewer | 检查职责范围内的正确性，不代替 Owner 重写全文 |
| coOwner | 只在明确章节或 Owner 不可达时代行，不等于多人共同拥有全文 |
| SM | 暴露冲突、推动确认，不负责代写或手工合并专业内容 |

铁律：一份正式文档只有一个整体 Owner。确需多人写时，按章节、表格或文件
拆分责任，而不是写 `owner: [A, B]`。

## 4. 最小评审闭环

```text
Owner 提交可消费版本
  -> Reviewer 只给“通过 / 修改项”
  -> Owner 处理并回复
  -> 任务表记录结论和证据
```

评审最多指定 1-2 个真正有决策责任的人。意见超过一个工作日未处理时，
SM 只提醒 Owner 和 Reviewer 对齐，不召集全员审批。

L1/L2 可选元数据：

```yaml
---
owner: TL
reviewers: [PO, FS]
status: draft
version: 0.1
last-updated: YYYY-MM-DD
---
```

L0 文件不要求添加这些字段。已有 Frontmatter 可以保留，不要求普通执行成员
为了改一行状态维护版本号。

## 5. 工具无关

团队可以使用 GitHub、GitLab、Codeup、共享文档或本地 Git，但工具不是准入门：

| 条件 | 做法 |
| --- | --- |
| 有 PR/MR | Owner 提交，Reviewer 在线确认 |
| 没有 PR 权限 | Owner 提交 patch/文件，Reviewer 在任务表确认 |
| CODEOWNERS 可用 | 仅用于自动通知，不作为能否开工的条件 |
| 平台暂不可用 | 保留本地 commit/文件和验证说明，恢复后补同步 |

`.github/CODEOWNERS` 是可选配置。占位账号未替换时不应阻塞 Sprint 0 或编码，
只代表自动指派尚未启用。

## 6. 高冲突时的结构选择

优先顺序：

1. 判断内容是否可以删除或改为任务表链接。
2. 按稳定边界拆成小文件。
3. 仍需同文档协作时，按章节或表格行指定 Owner。
4. 最后才启用强制 PR、CODEOWNERS 和自动检查。

总表等结构化文件优先按职责分表或行 Owner；设计文档优先单 Owner 汇总；
评审者写修改项，不在 Owner 尚未确认时直接改写结论。

## 7. 冲突处理

1. 停止继续编辑冲突区。
2. 确认主事实源和整体 Owner。
3. 各参与者提交自己的事实或建议，不互相覆盖。
4. Owner 按决策权合并；涉及价值、技术、发布分别交还 PO、TL、FS。
5. 在任务表记录最终结论，不再复制一份“冲突复盘报告”。

## 8. 通用独立评审输入机制

适用于任何需要多人评审的文档，不限于 Sprint Review/Retro：ADR、API 契约、
架构方案、测试策略、发布 Runbook、流程规范、会议纪要等都按同一原则处理。

核心规则：**正文只有 Owner 可维护；Reviewer 不直接改正文，只写独立评审输入；
Owner 采纳后汇总。**

### 8.1 存放结构

优先在被评审文档所在目录创建 `_review-inputs/`，按目标文档建子目录：

```text
<doc-dir>/
  <target-doc>.md
  _review-inputs/
    <target-doc-id-or-slug>/
      <member-id>.md
```

Sprint Review/Retro 可使用 Sprint 目录统一入口：

```text
03_迭代运行/Sprint-X/
  02_SprintX_Review纪要.md
  03_SprintX_Retro纪要.md
  _review-inputs/
    review/<member-id>.md
    retro/<member-id>.md
```

### 8.2 个人评审输入模板

```markdown
# <目标文档> · <成员> 评审意见

- 结论：approve / request-changes / comment
- 评审范围：我实际看过的章节、任务、代码范围或证据
- 必须修改：阻止通过的问题；没有则写“无”
- 可选建议：不阻止通过的优化建议
- 证据：commit、PR/MR、命令结果、截图或会议结论链接
```

### 8.3 职责

| 角色 | 动作 |
| --- | --- |
| Owner | 创建目标文档、列出 Reviewer、维护正文、采纳/拒绝意见、发布结论 |
| Reviewer | 只编辑自己的输入文件，不改正文、不替他人写意见 |
| SM | 检查 required reviewer 是否都有输入或显式豁免，提醒 Owner 汇总，不代替专业 Owner 裁决 |

### 8.4 闭环条件

文档进入 `approved` / `locked` 前至少满足：

1. required reviewer 均有独立输入文件，或在目标文档中写明豁免原因。
2. Owner 在目标文档写出“评审汇总/采纳结论”，链接个人输入文件。
3. 任务表或 06 总表写回状态和证据。
4. 若意见要求修改，Owner 修改后记录处理结论；不把未处理意见静默丢弃。

## 9. Review/Retro 追加兼容规则

新建 Review/Retro 默认采用 §8 的独立评审输入：成员写
`_review-inputs/review/<member-id>.md` 或 `_review-inputs/retro/<member-id>.md`，
SM 维护主纪要，只汇总结论、完整名单、改进项和链接。

`<!-- append-policy: anchors-v1 -->` 与成对 `append:role` 锚点仅作为历史文件或低争用
场景的兼容方案。出现以下任一情况，立即切换到独立评审输入：

- 两人以上可能同时编辑同一文档。
- 出现过重复标题、嵌套锚点、覆盖他人意见或误删。
- Review/Retro 需要全员追加长段意见。
- 某个正式文档需要 2 名以上 reviewer 并行给意见。

历史文件不追溯拆分；新 Sprint 和新关键文档优先使用独立输入文件，减少 blame
追溯和手工冲突修复。

签核不再使用多人追加表：每个 Event 独立为
`.team/signoffs/events/<campaign>/<event>.json`，天然消除同一 Markdown 表的
并发覆盖。Event 文件只允许一次创建提交；需要纠正时新建 Event，不修改旧文件。
若多人共用同一 worktree，`signoff.mjs` 还会在 Git common-dir 获取短事务锁，
串行化 add/commit；这只保护签核命令，不允许签核期间并发执行其他手工 Git 写操作。

## 10. 反模式

| 反模式 | 修正 |
| --- | --- |
| 多人评审直接追加到同一正文 | 拆为每人独立评审输入文件，主文档只由 Owner/SM 汇总 |
| Retro/Review 多人改同一块，覆盖他人标题/归属 | 新文档默认用独立评审输入；锚点仅作历史兼容 |
| 所有 Markdown 强制 Frontmatter + PR | 只对 L1/L2 启用 |
| Reviewer 直接重写 Owner 正文 | 写入自己的 `_review-inputs/<target>/<member>.md`，由 Owner 消化 |
| 为避免冲突不断拆文件 | 先检查内容是否值得保留 |
| SM 成为所有文档合并人 | 恢复专业 Owner |
| GitHub 权限未配置就全员等待 | 使用本地证据与人工确认继续推进 |
| 评审人越多越安全 | 限 1-2 名真正决策者 |

衡量本规范是否有效，不看文档数量或 PR 数，而看冲突是否减少、决策是否可
追溯，以及成员是否能更快回到价值交付。

## 附录 A · L2 合并协议模板

> 仅当文档升级到 L2（多人频繁并发修改同一正式产物）时才启用。
> 复制本附录到文档头部或紧邻文件，按实际内容填写。

### A.1 分区边界

按内容边界指定 Owner，每个分区独立编辑、互不覆盖：

| 分区（章节/表格区/文件） | Owner | 协作人 | 说明 |
| --- | --- | --- | --- |
| 待填写 | 待填写 | 待填写 | 例：§1-§3 = TL，§4-§5 = FE |

### A.2 合并窗口与节奏

| 项目 | 约定 |
| --- | --- |
| 合并频率 | 例：每日站会后 / 每次 Review 前 / 按需 |
| 合并顺序 | 例：先 rebase 再合并；冲突时 Owner 裁决 |
| 锁定期 | 例：Release 前 24h 冻结结论区，只允许勘误 |

### A.3 冲突仲裁

| 触发条件 | 仲裁人 | 时限 |
| --- | --- | --- |
| 同一分区两人同时编辑 | 该分区 Owner | 1 个工作日内 |
| 跨分区结论矛盾 | 整体 Owner 或 SM | 2 个工作日内 |
| 仲裁超时 | SM 升级到 Retro | — |

### A.4 回滚策略

| 场景 | 做法 |
| --- | --- |
| 合并后发现结论错误 | Owner 回退该 commit，修正后重新提交 |
| 分区 Owner 长期缺席 | SM 临时指派 coOwner，不改分区边界 |
| 冲突频发无法收敛 | 评估是否拆为独立文件（回到 §6 优先序 2） |

### A.5 降级条件

满足以下全部条件时，SM 可将文档从 L2 降回 L1：

- 最近 2 个 Sprint 无真实冲突。
- 分区 Owner 收敛为 1 人。
- 文档进入 locked 或稳定状态。
