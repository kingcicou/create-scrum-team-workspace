# 经验教训与反模式清单

## 1. QFD_Ark Sprint 0-2 关键教训

| 经验 | 说明 |
| --- | --- |
| 规范先于产出 | 产出目录、命名、总表规则要在第一批产物前确认 |
| 总表不是事后填 | 总表和产出同步，否则团队会误判进度 |
| 存量项目先诊断 | 成熟系统不要套从零项目的搭建流程 |
| 目录必须立即可用 | 规范引用的目录必须真实存在 |
| 命名前置 | 文件命名不统一会消耗检索和自动化效率 |
| 可执行代码和描述文档分离 | 脚本、CI、迁移、测试进仓库；结论和决策进工作区 |
| Draft 需要暂存区 | 未评审产物不要进入长期目录 |
| 规范要全员反馈 | 反馈、裁决、定版比单向下发更容易落地 |
| 模板化不能只抄工程骨架 | 05/06/SM/09 是团队协作神经系统 |
| 时间盒结束不等于全部Done | 关闭、目标、治理和下一轮准入必须分开 |
| 方案approved不等于门禁passed | 分层记录方案、门禁与交付结论 |
| 统计必须来自证据清单 | 手工汇总容易重复、漏项和事后改口径 |
| 关闭后必须切换当前入口 | tag 已创建但首页、日历和角色入口仍显示执行中，会误导下一轮 |
| 计划周期不能被结果日期覆盖 | 另记工作完成和正式关闭，用于容量校准和治理闭环 |
| CI 证据统一语义而非平台字段 | 平台没有统一运行 ID 时可用 Job ID、日志或哈希，但必须关联提交、时间、结果和验证人 |
| 稳定 CI 不必每个 Sprint 重建 | 仓库、技术栈、门禁或平台变化时才建立配置/迁移 Task |
| 测试数量不代表测试稳定 | 只报告实际命令、commit、结果和失败明细，不从文件数推断质量 |
| SM 的通知也是交付物 | 状态、签核或纠偏通知必须给出范围、动作、证据和闭环方式 |

## 2. 常见 Scrum 反模式

| 反模式 | 后果 | 修正 |
| --- | --- | --- |
| Scrum 只剩会议 | 团队疲惫但价值不清 | 围绕工件和证据重建流程 |
| PO 只传话 | Backlog 没有价值排序 | PO 必须做取舍 |
| SM 变项目经理 | 团队被动汇报 | SM 只清障和引导 |
| TL 只画架构图 | 设计不能落地 | ADR/API/拆分必须进入 Story |
| QA 末期集中测试 | 缺陷堆积 | AC 澄清时质量左移 |
| DevOps 隐形工作 | 容量失真 | DevOps 任务进入 Backlog |
| Review 展示 PPT | 没有真实反馈 | 展示可运行增量 |
| Retro 没有闭环 | 同样问题反复出现 | 1-3 个改进项进入台账 |

## 3. 模板演进反模式

| 反模式 | 风险 | 修正 |
| --- | --- | --- |
| 只生成目录 | 看起来完整，实际不会运行 | 生成 05/06/SM/TeamWork 机制 |
| 代码仓库维护第二套 docs | 双写、冲突、过期 | 外层工作区是文档主事实源 |
| 每个任务都产出文档 | 执行时间被写作和评审吞噬 | 普通任务只更新状态和证据 |
| 把高级治理设成默认 | Frontmatter、PR、审批成为开工门槛 | 按冲突风险从 L0 升级 |
| 外部工具不可用就停工 | 可并行实现也被无谓冻结 | 先实现并提交最小验证说明 |
| 资深成员只评审不拆分 | 中初级成员拿到模糊大任务 | 先拆模块、标复杂度和验收证据 |
| TeamWork 只写在 README | 团队执行时忘记 | 同步写入 `.gitignore`、总表、SM 检查 |
| 不区分项目类型 | 存量项目被迫从零搭建 | CLI 和 Sprint 0 模板区分类型 |
| 知识库承接当期执行 | 知识库变垃圾场 | 执行在工作区，稳定经验进知识库 |
| Sprint close tag被当成release | 未完成风险被掩盖 | tag消息明确Goal、证据、门禁和carry-over |
| tag 后首页和日历仍是执行中 | 团队无法判断当前 Sprint | 关闭清单强制切换入口并锁定事实 |
| 为了“准确”回写计划日期 | 丢失估算与实际偏差 | 保留计划周期，另记完成和关闭时间 |
| 不同 CI 平台字段不一致就判无证据 | 已验证工作被反复补材料 | 使用平台无关最小证据契约 |
| 每轮固定重建流水线 | 技术栈未变也消耗容量 | 变化触发；稳定流水线只做风险验证 |
| 用测试文件数或手工计数宣称稳定 | 统计无法复算，掩盖失败用例 | 保存实际命令、结果和失败明细 |
| SM 把签核编排转给开发成员 | 误派、漏派且职责倒置 | SM 建批次、发通知、核验；成员只追加本人事件 |
| 签核只保存最新基线 | 补签覆盖范围和历史时间不可追溯 | Change、Campaign、Event 分层记录 |
| 签核表手填自身 commit | 自引用导致二次回填，甚至填入代码仓 hash | Event ID 由 Git 自动反查提交 |
| 全量重基线抹平旧历史 | 当前有效被误写成过去完整 | 分开报告当前有效性与历史完整性 |
| 签核证据用 pickaxe（`git log -S`）反查 | 预铺的空行被归因给脚手架作者，代提不可见 | 用 `git blame` 取签核行当前作者，与成员列比对 |
| 异常（疑似代签/无效）事件仍计入覆盖 | 旧代签占位+新本人补签伪装“当前有效” | 只非⚠️证据计入验证覆盖，异常 CHG 显示待重签 |
| 把署名一致性检测当作不可伪造的身份认证 | 姓名大小写/显示名差异可绕过 | 定位为“Git 署名一致性检测”，可扩 author-mail 共同校验 |
| 新 `auto` 事件未提交仍计入当前覆盖 | 后续有效事件可能掩盖待提交证据 | 仅 ✅ 或兼容的 LEGACY unverified 计入；新事件待提交单列 |
| Campaign 已关闭但审计仍有待处理 | “closed”与实际签核事实矛盾 | 保留旧批次，SM 建 corrective 批次直至归零 |
| SOP 要求检查 Review/Retro，但默认文件没有追加区 | 新项目首次运行工具即失败 | 默认提供空追加区，有条目才要求成对锚点 |
| 他人预铺 Event，本人改空格“认领” blame | 当前作者看似正确，但事件内容并非本人创建 | 每个 Event 独立文件；首次作者+邮箱匹配且创建后不可修改 |
| 通知写全局范围，示例却固定 CHG-150 | 成员复制示例造成覆盖不足 | Campaign 文件保存逐角色 assignment，sign 自动读取 |
| 依据“已有 n 行”关闭 Campaign | 行数替代覆盖和证据审计 | close 仅 SM 可执行且 status 未归零时返回失败 |
| Campaign 内 7/7 有效就当作项目全局完成 | 局部覆盖掩盖其他历史缺口 | 生成机器审计 JSON；close 同时要求全局待处理归零 |
| 多角色共用目录时轮流改 Git 身份 | 配置竞争、普通提交串号、index.lock 冲突 | 命令级身份不落配置；签核写操作使用仓库互斥锁 |
| 通知字段只存在于 SM 的临时文案 | 范围、阅读要求和截止无法复现 | Campaign V2 结构化保存，notify 直接生成可转发通知 |
| 错误范围能生成通知，到 close 才失败 | 全员完成无效动作后才发现输入错误 | Campaign V3 保存来源指纹；prepare/notify/sign 前置 verify |
| 在脏工作区生成 Campaign | 未提交事实进入审计，推送后同一 HEAD 无法复现 | 审计前要求干净事实源；保存 HEAD/tree；notify 要求 exact=yes |
| 用 `cat-file` 证明 legacy 签核 | 悬空对象只在某个本地仓库存在，干净克隆审计不同 | legacy commit 必须是当前 HEAD 的祖先 |
| Campaign 正确后手工压缩通知 | 区间表达、合并角色、补写截止导致群聊与事实源分叉 | prepare 强制 due；publish 提交 Notice；sign 校验摘要 |

## 4. Sprint 结束后的知识沉淀问题

每轮 Retro 后，SM 可以问：

- 这个问题是否重复出现？
- 这个经验是否对下个项目有价值？
- 它是流程规则、角色协同、工程规范，还是工具手册？
- 是否已经在 05/06/SM/09 或知识库中有位置？
- 需要新增知识，还是修正现有知识？
