# 敏捷流程与工件指南

## 1. Scrum 的核心对象

| 对象 | 含义 | 常见误区 |
| --- | --- | --- |
| Product Backlog | 产品所有待办的有序列表 | 当成任务清单，没有价值排序 |
| Sprint Backlog | 当前 Sprint 承诺围绕 Sprint Goal 完成的工作 | 被动派活，没有团队共创 |
| Increment | 可运行、可验证、可演示的增量 | 只有代码写完，没有验收证据 |
| DoR | Story 进入 Sprint 前的就绪条件 | Planning 才开始澄清需求 |
| DoD | Story 标 Done 前的完成条件 | 把“本地跑通”当 Done |

## 2. 标准事件

| 事件 | 目的 | 输出 |
| --- | --- | --- |
| Refinement | 澄清、拆分、估算候选需求 | Ready Story、待澄清问题 |
| Sprint Planning | 确定 Sprint Goal、范围、容量、风险 | Sprint 计划、Sprint Backlog |
| Daily Scrum | 同步围绕 Sprint Goal 的协作计划 | 今日调整、障碍更新 |
| Sprint Review | 展示真实增量，收集反馈 | 反馈分类、Backlog 调整 |
| Retrospective | 改进团队协作系统 | 1-3 个改进项 |

## 3. 工件链

| 阶段 | 输入 | 输出 | 存放位置 |
| --- | --- | --- | --- |
| 机会发现 | 用户反馈、业务目标 | 愿景、场景、机会取舍 | `01_产品发现/` |
| Backlog 建立 | 愿景、路线图 | Epic、Story、AC | `02_产品待办/` |
| 迭代计划 | Ready Story、容量 | Sprint Goal、风险、分工 | `03_迭代运行/` |
| 工程设计 | Story、AC、约束 | ADR、API、数据模型、Spike | `04_工程设计/` |
| 实现验证 | Story 工作区、分支 | PR、CI、测试、验收证据 | 代码仓库 + `05_质量验证/` |
| 发布复盘 | Done 增量、质量结论 | Release Notes、Retro 改进项 | `06_发布运维/`、`07_度量改进/` |

## 4. DoR 建议

- Story 有明确用户价值或工程风险说明。
- AC 可验证，包含边界条件和失败场景。
- 依赖、风险和假设已显性化。
- 大小能在一个 Sprint 内完成，或已拆分。
- 涉及并行开发时，API/设计/数据草案已可用。
- 团队完成估算，不是单人拍脑袋。

## 5. DoD 建议

- AC 全部满足。
- 代码已 Review。
- CI 或本地等价验证通过。
- 测试证据已记录。
- PO 或授权验收人已确认。
- Story 工作区记录分支、PR、CI、测试、验收和 FS 集成结论。
- 涉及发布时，发布/回滚说明已补齐。

## 6. 插单规则

| 场景 | 处理 |
| --- | --- |
| 不影响 Sprint Goal | 进入 Product Backlog，参加下次 Refinement |
| 影响 Sprint Goal 但不紧急 | PO 记录，Review 后排序 |
| 必须本 Sprint 处理 | PO 明确替换范围，不无声追加 |
| 线上事故/安全风险 | 触发事件处理，事后 Retro 复盘 |

## 7. 质量左移

- AC 澄清时同步测试点。
- API 契约先于前后端并行。
- 测试脚本和开发同步演进。
- 缺陷进入缺陷分析，不靠聊天记录追踪。
- Release Go/No-Go 需要质量结论，而不是“应该没问题”。
