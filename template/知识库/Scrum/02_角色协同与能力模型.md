# 角色协同与能力模型

## 1. 小团队多帽子原则

本模板面向 5-9 人精简 Scrum 团队。一个成员可以戴多顶帽子，但每顶帽子的责任必须清楚。

| 原则 | 说明 |
| --- | --- |
| 帽子显性化 | 谁临时承担 QA、UX、DevOps，要写入总表和 Backlog |
| 能力备份 | 每个关键帽子至少有兜底人 |
| 隐形工作进 Backlog | CI、环境、设计系统、技术债都要占容量 |
| 不用角色压人 | 角色是协作接口，不是考核标签 |

## 2. 推荐角色槽位

| 槽位 | 主要职责 | 反模式 |
| --- | --- | --- |
| PO / PM | 愿景、价值排序、Backlog、AC、验收、发布取舍 | 需求传声筒、许愿池 |
| SM | 节奏、清障、总表、Retro、流程健康 | 派活、催进度、做考核 |
| TL / Sr.BE | 架构演进、ADR、关键 PR、后端核心风险 | 只画图不落地 |
| Mid.BE / QA API | 业务接口、服务逻辑、接口/集成测试 | 本地跑通就提测 |
| Sr.FE / UX | 前端架构、体验、设计系统、可访问性 | 只切图不管体验 |
| Mid.FE / QA E2E | 页面组件、联调、前端测试、视觉回归 | 提交后破坏基线 |
| FS / DevOps | 端到端闭环、CI/CD、环境、发布、集成分支 | 手工救火不自动化 |

## 3. 能力矩阵

| 成员 | BE | FE | QA | DevOps | 架构 | UX/UI | 说明 |
| --- | :-: | :-: | :-: | :-: | :-: | :-: | --- |
| PO | - | - | - | - | - | - | 价值和验收 |
| SM | - | - | - | - | - | - | 流程和清障 |
| TL/Sr.BE | 高 | 中 | 中 | 中 | 高 | 低 | 架构兜底 |
| Mid.BE | 中 | 低 | 中 | 低 | 低 | 低 | 接口和测试 |
| Sr.FE/UX | 低 | 高 | 中 | 低 | 中 | 高 | 体验和前端架构 |
| Mid.FE | 低 | 中 | 中 | 低 | 低 | 中 | 页面和 E2E |
| FS/DevOps | 中 | 中 | 中 | 高 | 中 | 低 | 集成和发布 |

## 4. 协作交接

| 交接 | 输入 | 输出 |
| --- | --- | --- |
| PO -> 团队 | Story、AC、价值、边界 | DoR 判断 |
| TL -> BE/FE | 架构、API、数据模型、拆分建议 | 可并行任务 |
| UX -> FE | 交互、组件、状态、可访问性要求 | 页面/组件实现 |
| BE -> FE | API 契约、Mock、错误码 | 联调基础 |
| QA -> 全员 | 测试点、缺陷、质量结论 | DoD 证据 |
| FS -> 全员 | 分支、环境、CI、部署 | 可运行增量 |
| SM -> 全员 | 节奏、障碍、总表、改进项 | 协作闭环 |

## 5. 心得沉淀

来自团队心得的共同观点：

- PO 不是写需求的人，而是守住价值和取舍的人。
- SM 守护目的，不守护仪式。
- TL 需要把技术未知转成 Spike 或 ADR。
- QA 是质量左移，不是末期验尸。
- UX 是质量的一部分，不是美化阶段。
- FE/BE 都要对测试和可演示增量负责。
- FS/DevOps 不应靠手工救火，要沉淀自动化和 Runbook。
