# Git 仓库布局与提交模式解析

> 这份文档回答"为什么这样设计"和"什么时候选 A 而不选 B"。
> 操作步骤见 `00_项目导航/08_团队开发协作SOP.md`。

## 1. 为什么有两层：项目工作区 + 代码仓库

| 层 | 位置 | 内容 | Git 角色 |
| --- | --- | --- | --- |
| 项目工作区 | `<project>/` 下 `00_~07_、90_、知识库/` | 文档主事实源 | 文档版本控制 |
| 代码仓库 | `<project>/10_代码仓库/<RepoName>/` | 可执行资产 `apps/`、`infra/`、CI | 代码版本控制 |

**为什么不合一**：

| 维度 | 文档主事实源 | 代码仓库 |
| --- | --- | --- |
| 更新频率 | 每日，全员 | 每日，主要编码角色 |
| 评审机制 | 文档 Review，PO/SM 主导 | Code Review，TL/FS 主导 |
| 受众 | 团队内部 + 利益相关方 | 团队内部 + 可能的外部贡献者 |
| 体积 | 小（KB-MB 级） | 大（含构建产物、依赖、二进制） |
| 公开性 | 通常内部 | 可能开源 |

**为什么不分两个文件夹放在不同位置**：保持物理相邻，让"决策 → 实现"的链路可追溯。开发者打开同一个 VS Code 工作区就能看到产品决策、AC、架构、测试和代码。

## 2. 三种 Git 模式取舍

| 模式 | 外层 .git | 内层 .git | 用一句话描述 |
| --- | :-: | :-: | --- |
| **A. workspace** | ✅ | ❌ | 文档+代码同仓，最简单 |
| **B. repo**（生成器默认） | ❌ | ✅ | 先独立代码仓，文档可随后独立管理 |
| **C. 两仓独立** | ✅ | ✅ | 文档仓 + 代码仓物理嵌套、逻辑解耦 |

### 模式 A — workspace

- **优点**：克隆一次拿全部、PR 同时改文档和代码、新成员入门门槛最低。
- **缺点**：代码仓不能独立 push 到外部 CI、不能给非文档团队的人 clone、push 量含大量文档变更。
- **适用**：Sprint 0 / 早期 / 内部项目 / 团队 ≤ 7 人。

### 模式 B — repo

- **优点**：代码仓清爽。
- **缺点**：文档变更没有 git 历史可追溯。
- **适用**：罕见。一般作为模式 C 的过渡形态。

### 模式 C — 两仓独立（生产推荐）

- **优点**：代码仓和文档仓各自有独立的 GitHub 仓库、权限、CI、PR 流程。代码仓可开源/可分发给外部协作者，文档仓保留内部 Backlog/Sprint 细节。
- **缺点**：克隆和初始化步骤多；文档 PR 与代码 PR 不能在同一个 PR 里完成。
- **适用**：产品上线后 / 团队 > 7 人 / 有外部协作 / 代码需要独立 CI 凭证。

### 选型决策树

```text
启动期？ ──── 是 ──> 模式 A
   │
   否
   │
代码需要独立访问权限/CI/对外发布？ ──── 是 ──> 模式 C
   │
   否
   │
保持模式 A
```

## 3. `git worktree` 为什么不是"独立仓库"

源自 git 2.5+ 的特性：**多个工作目录共享同一个 `.git` 对象库**。

```text
代码仓 .git/             ← 一份对象库
├── 主工作目录            ← checkout main
├── TeamWork/Evan_xx     ← checkout feature/sprint-1/login-evan
├── TeamWork/Ritchie_xx  ← checkout feature/sprint-1/api-ritchie
└── TeamWork/Fowler_xx   ← checkout feature/sprint-1/arch-fowler
```

| 维度 | 说明 |
| --- | --- |
| 物理 | 多目录，互不影响 |
| 逻辑 | 同一对象库，commit/branch 自动可见 |
| 远程 | 任一 worktree 推送 = 同一远程 |
| 跟踪 | `TeamWork/` 已 `.gitignore`，主仓库不跟踪其存在 |
| 替代品 | 也可以多次 `git clone`，但浪费磁盘且分支同步麻烦 |

### 为什么不让每个角色独自 `git clone`

- 浪费磁盘（对象库重复）
- fetch/push 节奏割裂，sprint-x 上的代码新鲜度不一致
- worktree 的"轻量分身"语义更符合"团队协同"

## 4. 何时应当把代码仓独立成 Git（A → C 升级）

触发任一条件，就该升级：

- [ ] 代码仓需要 push 到与文档仓不同的 GitHub/GitLab
- [ ] 代码仓有外部贡献者或要公开开源
- [ ] 代码仓需要独立 CI 凭证、独立 secrets
- [ ] 代码仓的访问权限严于文档仓
- [ ] 文档仓体积过大或频繁推送拖慢编码角色

**升级步骤**见 `00_项目导航/08_团队开发协作SOP.md` §6.7。

## 5. 提交身份追溯的设计

铁律：**每个 commit 都能追溯到具体真实的人**。

| 风险 | 防御机制 |
| --- | --- |
| 使用统一账号代提 | 禁止；启用 `extensions.worktreeConfig` 后，每个 worktree 使用 `git config --worktree` |
| 自动化机器人提交 | 必须用 bot 专属账号（如 `ci-bot`），名字明确 |
| 多人共用一台机器 | 每人独立 worktree + 独立身份配置 |
| 邮箱写错 | 后续 commit 改对即可；不强制改写已 push 的历史 |

工具：`git log --pretty='%an <%ae>'` 可审计身份。

## 6. 与"两仓"配套的分支模型

无论模式 A/C，分支模型一致：

| 分支 | 责任 | 生命周期 |
| --- | --- | --- |
| `main` | FS 守护 | 长期，只接收已验证增量 |
| `sprint-<n>` | FS 建并维护 | 一个 Sprint |
| `feature/sprint-<n>/<topic>-<name>-<role>` | 个人 | 1-3 天 |
| `hotfix/<issue>` | TL/FS | 紧急修复 |

合并策略：

```text
个人特性分支 ── PR ──> sprint-x ── 集成测试通过 ──> main ── 打 tag ──> 发布
```

## 7. 本次迭代的 FAQ（释疑归档）

### Q1：为什么生成器默认用模式 B，而不是直接创建两仓？

A：角色 worktree 只需检出代码，默认独立代码仓可避免重复五份项目文档。文档需要版本控制时，再由 SM/FS 初始化外层文档仓并升级为模式 C。决策依据见 `知识库/项目模板/03_角色工作区自动化迭代计划.md`。

### Q2：模式 A 下 `10_代码仓库/<repo>/.gitignore` 为什么还要存在？

A：为模式切换做准备。即使外层 git 不区分边界，子目录的 `.gitignore` 在升级到模式 C 时**直接可用**。提前定义 = 减少未来变更面。

### Q3：`TeamWork/` 为什么用三个备选命名（TeamWork/SprintWork/DevWorktrees）？

A：团队习惯不同，统一硬编码会引起摩擦。**推荐 TeamWork/**（规范默认），`.gitignore` 把三个都列入是**容错**，团队选了别的名字也不会误提交。

### Q4：worktree 和"独立 clone"哪个更好？

A：**优先 worktree**。共享对象库节省磁盘、分支同步更直接。但 `git worktree` 对老版 git（<2.5）不可用，遇到老环境用 clone 兜底。

### Q5：文档变更要不要走 sprint-x？

A：**不需要**。文档 PR 通常直接对 `main`，由 PO/SM Review。涉及 AC/Backlog 重大改动才需要 PO 评审。代码变更必须经 sprint-x，因为要 FS 集成 + CI 验证。

### Q6：模板自身的 `create-scrum-team-workspace` 仓库为什么独立于 QFD_Ark？

A：模板是**复用资产**，会被多个项目消费；QFD_Ark 是**应用项目**，是模板的首个消费者。二者生命周期、贡献者、版本节奏完全不同，所以独立 git + 独立 GitHub 仓库。这就是模式 C 在"模板/项目"层级的应用。

### Q7：QFD_Ark 主仓的 `.gitignore` 为什么排除 `10_代码仓库/create-scrum-team-workspace/`？

A：避免 git 嵌套警告与双重历史。物理上文件还在工作区里，**逻辑上**由独立的 GitHub 仓库 `kingcicou/create-scrum-team-workspace` 维护。这是模式 C 的具体落地。

### Q8：模式 A 下，"统一文档+代码 push"会不会让 PR 太大？

A：**会**，但可用以下手段缓解：
- 文档 PR 走 main，代码 PR 走 sprint-x，**物理上分开提**
- 提交分粒度小，文档 commit 和代码 commit 分开
- PR 大到 Review 困难时，立即拆分

如果团队规模/PR 频率让模式 A 不再合适，就是 §4 触发的 A→C 升级时机。

### Q9：为什么个人分支不能叫 `sprint-1/...`？

A：Git 引用以路径存储，`sprint-1` 分支与 `sprint-1/...` 分支存在文件/目录冲突，无法同时创建。集成分支保留 `sprint-1`，个人分支使用 `feature/sprint-1/...`。
