# 角色快速上手与成长指南

> 新成员不需要通读知识库。先从项目首页找到当前 Sprint 和自己的 Task，
> 能开始交付后，再按问题查阅一份指南。

## 1. 唯一上手入口

先打开 `00_项目导航/00_项目首页.md`。首页必须明确链接当前 Sprint，不能让
成员根据目录编号猜测“哪个数字最大就是当前 Sprint”。

从首页按以下顺序打开：

| 顺序 | 事实 | 模板中的初始位置 | 要找到什么 |
| --- | --- | --- | --- |
| 1 | Sprint Goal | `03_迭代运行/Sprint-0-启动/00_Sprint计划.md` | 本轮为何做、边界是什么 |
| 2 | Sprint 任务表 | `03_迭代运行/Sprint-0-启动/01_Sprint任务表与流程看板.md` §4 | 自己的 Task、Owner、前置、Reviewer、证据 |
| 3 | Story / AC | `02_产品待办/02_Stories/Sprint1_候选Backlog.md`、`02_产品待办/03_AC_验收标准/README.md` | 用户价值和可验证条件 |
| 4 | 仓库清单 | `10_代码仓库/00_仓库清单.md` | 当前开发仓、基线分支、仓库策略 |
| 5 | 代码与证据 | 仓库清单指向的代码仓、PR、CI 或测试结果 | 实际交付位置 |

后续 Sprint 的目录名会变化，但首页和文件名约定不变：
`03_迭代运行/<当前Sprint>/01_Sprint任务表与流程看板.md`。

## 2. 先分清六个对象

| 对象 | 回答的问题 | 是否通常单独建文档 |
| --- | --- | :---: |
| Goal | 本轮为什么值得做 | 写入 Sprint 计划 |
| Story | 用户/业务要获得什么价值 | 写入 Backlog；不必每条另建工作区 |
| AC | 怎样算 Story 被验收 | 写入 Story 或 AC 事实源 |
| Task | 谁为 Story 做哪项可执行工作 | 写入 Sprint 任务表 |
| 模块 | 任务影响的功能、页面、服务、数据库或基础设施边界 | 否；它是拆分概念，不是固定目录 |
| 证据 | 什么能证明 Task/AC 已完成 | 代码提交、PR、测试、截图或结论链接 |

`06_团队输入输出总表/` 是跨 Sprint 的长期交付账本，只登记需要长期追踪的
契约、决策、报告和发布产物。普通 Task 只更新 Sprint 任务表，不再抄到总表。

## 3. 30 分钟开始交付

| 时间 | 动作 | 完成信号 |
| --- | --- | --- |
| 0-5 分钟 | 从首页确认当前 Sprint 和 Goal | 能说清本轮目标 |
| 5-10 分钟 | 在任务表 §4 按姓名/角色查找 | 找到 Task ID、状态、前置和 Reviewer |
| 10-15 分钟 | 沿 Story/AC 链接确认输入 | 知道做什么和怎样验收 |
| 15-20 分钟 | 从仓库清单确认代码仓、分支和验证命令 | 不会提交到错误仓库 |
| 20-25 分钟 | 与 Reviewer 确认边界、证据和第一个小步骤 | 双方对 Done 理解一致 |
| 25-30 分钟 | 更新状态为进行中，开始第一个 S/M Task | 已开工或已准确报告等待/阻塞 |

环境暂缺但仍能编码时，继续可逆工作；由 FS 指定验证环境、验证人和补验时间。

## 4. 各角色的第一条路径

| 角色 | 首先打开 | 首项行动 |
| --- | --- | --- |
| PO | Sprint 计划 → Backlog/AC → 任务表 §4 | 确认 Goal、Story 价值、范围与验收边界 |
| SM | 首页 → 任务表 §2/§3/§4 → 风险清单 | 校准阶段、依赖、状态新鲜度和阻塞 |
| TL / Sr.BE | 任务表 §4 → 技术全景/ADR → Story/AC | 把 L/跨模块工作拆成可执行 Task，指定 Reviewer |
| Sr.FE / UX | 任务表 §4 → Story/AC → 前端设计约束 | 拆页面/交互切片，确认体验边界 |
| Mid.BE | 任务表中本人行 → Story/AC → API/数据契约 | 实现 S/M Task，提交测试证据 |
| Mid.FE | 任务表中本人行 → Story/AC → API/体验契约 | 实现页面/联调 Task，提交测试证据 |
| FS / DevOps | 仓库清单 → 协作 SOP §4 → 任务表中本人行 | 确认仓库、身份、分支、CI 或替代验证 |

普通执行角色不必先读 SM 原理、模板演进、文档治理或 Sprint 关闭规范。

## 5. 一个完整例子

```text
Goal：用户可登录系统
  └─ Story ARK-101：作为注册用户，我可以使用账号登录
      ├─ AC1：正确账号进入首页
      ├─ Task T04：实现登录 API（Mid.BE，前置=T03，Reviewer=TL）
      └─ Task T05：实现登录页（Mid.FE，前置=API 契约，Reviewer=Sr.FE）
           └─ 证据：提交/PR + 自动化测试或截图
```

T04 与 T05 可在契约明确后并行；T05 不必等待 T04 全部完成。成员只更新各自
Task 行，Story/AC 和长期总表不重复记录每日进度。

## 6. 状态怎么说

状态统一使用：`未开始`、`进行中`、`等待输入`、`阻塞`、`待评审`、`已完成`。

需要向 SM 或群聊同步时，用一条状态包：

```text
Task T04｜进行中｜已完成接口骨架｜等待 T03 数据模型｜
下一输出：登录 API + 单测｜预计：今天 17:00｜证据：<链接>｜SM：待确认
```

SM 回复“已同步”或指出待澄清字段。详细格式见
`00_项目导航/09_SM教练查询与回复模板.md`。

## 7. 谁负责带新人

| 内容 | 责任人 |
| --- | --- |
| 价值、Story、AC | PO |
| 架构边界、任务拆分、复杂度 | TL / 对应资深工程师 |
| 代码仓、分支、环境 | FS |
| 节奏、状态语言、依赖沟通 | SM |
| 具体 Task 走读与 Review | Task Reviewer |

SM 只确认上手链路没有断点，不替专业 Owner 讲解或补写交付物。

## 8. 上手完成与成长

新人能回答以下五问即可开始，不设置培训考试或读书打卡：

1. 当前 Sprint Goal 是什么？
2. 我的 Task ID、前置、Reviewer 和状态是什么？
3. 输入来自哪个 Story/AC/契约？
4. 输出放在哪里，用什么证据证明？
5. 等待或阻塞时通知谁，SM 是否已经同步？

| 阶段 | 行为证据 |
| --- | --- |
| L1 上手 | 独立完成 S Task，状态和证据准确 |
| L2 独立 | 完成 M Task，识别依赖并做有效 Review |
| L3 资深 | 拆分 L Task、降低不确定性、带新人 |
| L4 传承 | 将重复问题提炼为更短指南、自动化或模板改进 |

学习心得和培训材料默认可选。只有内容可复用且无法通过修改现有文档表达时，
才新增知识库文档。
