# 产品发现与价值排序指南

> 面向 PO/PM。Backlog 是结果，不是起点；起点是价值假设与证据。

## 1. 产品发现的三层证据

| 层级 | 问题 | 证据 |
| --- | --- | --- |
| 用户 | 谁在用？什么场景？什么痛点？ | 访谈、可用性观察、客服工单 |
| 价值 | 解决后他获得什么？规模多大？ | 场景成功率、留存、营收、节省成本 |
| 可行 | 能在多少 Sprint 内交付？风险何在？ | 技术 Spike、依赖盘点、合规清单 |

任一层缺证据，Story 不应进入 Sprint Planning。

## 2. 价值假设模板

```text
我们认为，对 <用户/角色>
在 <场景> 下
提供 <能力/改变>
能够带来 <可度量的结果>
我们将通过 <信号/指标> 验证它是否成立。
```

写在 `02_产品待办/01_Epics/` 对应 Epic 的开头。Sprint Review 用这条假设作为对照。

## 3. 优先级模型

| 模型 | 适用场景 | 取舍 |
| --- | --- | --- |
| Value vs Effort | 早期、机会多 | 简单但易高估价值 |
| RICE | 需要跨方案比较 | Reach × Impact × Confidence ÷ Effort |
| WSJF | 多依赖、多约束 | (业务价值+时间敏感+风险/机会) ÷ 工作量 |
| Kano | 体验类、留存类 | 区分基础/期望/兴奋型 |

模板默认不绑定单一模型。建议 PO 在 Sprint 0 选定一个并写入 `01_产品发现/05_机会评估与取舍记录.md`。

## 4. Story 拆分要点

- 沿用户路径切，不沿技术层切。
- 一个 Story 一个可演示结果。
- AC 含正常路径、边界、失败、空态、权限。
- 大型探索拆为 Spike，结果回填到 Story 的依赖里。

## 5. PO 与团队的协作时刻

| 时刻 | PO 动作 |
| --- | --- |
| Refinement | 带证据来，不带"老板说"来 |
| Planning | 守 Sprint Goal，不塞兜底愿望 |
| Daily | 不派活，只澄清需求与取舍 |
| Review | 看可运行增量，不看 PPT |
| Retro | 反思自己是否成为团队瓶颈 |

## 6. PO 反模式

| 反模式 | 后果 | 修正 |
| --- | --- | --- |
| 需求传声筒 | Backlog 无主线 | 用价值假设过滤需求 |
| 许愿池式 Backlog | 永远做不完 | 每 Sprint 主动砍 P3 以下 |
| AC 模糊 | 开发后期返工 | AC 必须可验证 |
| 临时插单常态化 | Sprint Goal 失效 | 启用插单规则 |
| 验收靠"差不多" | DoD 形同虚设 | 验收人按 AC 逐条对齐 |
| 路线图只在 PO 脑里 | 团队失向 | 路线图进 `01_产品发现/` |

## 7. 最小输出物

PO 在 Sprint 0 结束前应产出：

- [ ] 产品愿景与目标（`01_产品发现/00_产品愿景与目标.md`）
- [ ] 至少 1 个主用户画像和 2 个核心场景（`01_产品发现/01_用户画像与场景.md`）
- [ ] 1-3 条价值假设（`01_产品发现/02_价值假设.md`，每条可证伪、可度量）
- [ ] 竞品/替代方案对照 + 差异化定位一句话（`01_产品发现/03_竞品与市场.md`）
- [ ] 北极星指标 + 12 周路线图（`01_产品发现/04_路线图与北极星指标.md`）
- [ ] 机会评估与取舍决策日志（`01_产品发现/05_机会评估与取舍记录.md`）
- [ ] 首批 Epic 与候选 Backlog 满足 DoR（`02_产品待办/`）
- [ ] 价值排序模型已选定并公开
- [ ] 已确认验收机制与验收人
