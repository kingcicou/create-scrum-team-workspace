# Sprint 0 与项目类型指南

## 1. Sprint 0 的定义

Sprint 0 不是用来交付业务功能的 Sprint。它用于建立能持续交付的最小基座。

不同项目类型的 Sprint 0 完全不同，不能套一套固定模板。

## 2. 项目类型

| 类型 | 判断标准 | Sprint 0 重点 | 代码策略 |
| --- | --- | --- | --- |
| 从零新项目 | 没有可运行代码 | 愿景、首批 Backlog、架构草案、工程骨架 | 生成新主仓库 |
| 存量项目重构 | 有旧系统，目标升级治理 | 现状诊断、风险评估、迁移路线图 | 旧仓库只读或受控修改 |
| 成熟产品迭代 | 产品已运行，持续交付 | Backlog 梳理、容量规划、质量门禁 | 既有仓库按 Sprint 分支 |
| 原型转正 | Demo/POC 工程质量不足 | 技术选型复核、质量和安全基线 | 业务知识可复用，混乱代码不照搬 |

## 3. 从零项目 Sprint 0 门禁

- 产品愿景初版完成。
- 首批 Epic / Story / AC 建立。
- 工程骨架可启动。
- 技术全景和 ADR 候选明确。
- API 和数据模型有初稿。
- 质量策略和发布策略有底线。
- 团队协议、05 规范、06 总表已确认。

## 4. 存量项目 Sprint 0 门禁

- 参考源只读，结论写入项目工作区。
- 架构、API、数据、前端体验、质量、运维均完成诊断。
- P0 风险已转 Story / Spike / 技术债。
- 升级路线图和首批 Backlog 已形成。
- 明确哪些代码可复用、哪些重写、哪些暂不碰。
- 不在 Sprint 1 同时承诺大量新功能和大重构。

## 5. 常见教训

| 教训 | 说明 |
| --- | --- |
| 规范要先于产出 | 否则文件会散落到错误目录 |
| 目录必须立即可用 | 规范引用的目录要同步创建 |
| 总表必须同步 | 总表不同步，团队会误判状态 |
| 成熟项目先诊断 | 不要把存量项目当从零搭建 |
| 可执行代码和文档分离 | 脚本进仓库，结论进工作区 |
| 反馈机制很重要 | 规范要评审、裁决、定版 |
